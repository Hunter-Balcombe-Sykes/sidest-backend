## Plain English

When Shopify webhooks were being registered during app install, if one topic failed (e.g. due to a network blip), the job would quietly record it as "partial" and exit successfully — meaning it would never retry and the integration would be permanently incomplete. Now the job throws an exception when any topic fails, causing the queue to retry it up to 3 times with backoff. Webhooks that already registered are skipped on retry so there's no duplication.

## Technical Summary

**File changed:** `app/Jobs/Shopify/RegisterShopifyWebhooksJob.php`

**Changes:**
1. Narrowed the per-topic catch from `\Throwable` to `\RuntimeException`. `ShopifyClientException` (and its subclasses: `ShopifyGraphQLException`, `ShopifyTransportException`, `ShopifyThrottledException`) all extend `RuntimeException`, so they remain caught. Anything outside that hierarchy (e.g. `TypeError`, `Error`) now propagates immediately and fails the job cleanly rather than being silently swallowed.
2. After the topic loop, if `$allSucceeded === false`, a `RuntimeException` is thrown naming the failed topics. This causes the Laravel queue worker to retry the job (up to `$tries = 3` with backoff of 10/30/60 seconds). The `findExistingWebhook` check makes retries idempotent — already-registered topics are skipped, so only genuinely failed topics are reattempted.
3. Metadata is still written before the throw so there is a durable record of partial state if all retries are exhausted.

**New behavior contract:** Job succeeds (exits 0) only when every topic is either newly registered or confirmed already registered. Any per-topic failure that persists across all retries results in a failed job entry, surfacing the problem visibly rather than hiding it.
