## Plain English

The audit flagged that a shared signature-checking method in a parent class hadn't been reviewed. On inspection, both the Square and Fresha webhook controllers each have their own private `isValidSignature` method — there's nothing shared in the parent `ApiController`. Both methods correctly verify signatures using a timing-safe comparison and the raw request body, so no changes were needed. The audit item was downgraded from P1 to P3.

## Technical Summary

Verified `SquareCatalogWebhookController::isValidSignature` (line 124) and `FreshaCatalogWebhookController::isValidSignature` (line 141). Both are `private` methods on the respective controllers — not inherited from `ApiController`. Security posture confirmed:
- `hash_equals()` used for timing-safe comparison in both
- Raw body via `$request->getContent()` used in both (not JSON-decoded)
- HMAC-SHA256 algorithm used in both, matching Square's documented scheme
- Square additionally normalizes candidate notification URLs to handle URL variations
- Fresha's implementation carries a NOTE to re-verify against Fresha docs when the integration goes live (Fresha has no public booking API yet)
