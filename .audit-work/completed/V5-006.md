# #V5-006 — ProcessShopifyOrderWebhookJob filter races vs unique constraint, dropping commissions

## Plain English

When Shopify sends the same order webhook twice at nearly the same moment (which it sometimes does), two background workers could both check the database at the same time, both see "no entries yet," and then race to insert the commission records. PostgreSQL would let one worker win and force the other's entire insert batch to fail — but the failed worker's job would log "success" on retry with 0 entries created. This fix makes each row insertion independent: if one row already exists (because the other worker got there first), it's silently skipped, and the remaining rows are still created normally.

## Technical Summary

**File changed:** `app/Jobs/Shopify/ProcessShopifyOrderWebhookJob.php`

The Phase 2 insert block previously wrapped all `CommissionLedgerEntry::save()` calls in a single `DB::transaction()`. The pre-filter (`whereIn` on `idempotency_key`) ran outside that transaction, creating a TOCTOU window: two concurrent workers could both pass the pre-filter, then one's transaction would be aborted by a `23505` unique constraint violation when the other committed first. PostgreSQL aborts the entire transaction on any constraint violation, so no entries were created in the losing worker's batch even though the job ultimately reported success (on retry with 0 new candidates).

**Fix:** Removed `DB::transaction()`. Each `save()` now runs as its own implicit transaction. A `\Illuminate\Database\UniqueConstraintViolationException` catch per row silently skips keys that a concurrent worker already committed. The pre-filter remains as an optimization for clean re-deliveries (avoids any insert attempt for known-existing keys). Observer timing is unaffected: `CommissionLedgerEntryObserver` has `public bool $afterCommit = true`, so notifications already fired post-commit on each individual save.

Also removed the now-unused `use Illuminate\Support\Facades\DB` import and updated the Phase 2 comment to describe the new approach.

**Test added:** `tests/Feature/Jobs/Shopify/ProcessShopifyOrderWebhookJobNoLazyLoadTest.php` — new "is idempotent" test asserts that running the job twice with the same payload produces exactly 2 ledger entries (no duplicates, no exception).

## Decisions Made

- **No outer transaction:** Removed `DB::transaction()` entirely rather than using SAVEPOINTs/subtransactions. Each ledger entry is independently meaningful (one per line item); there's no business requirement that all entries for an order are atomic as a group. The pre-filter already violated all-or-nothing semantics anyway.
- **Kept the pre-filter:** It remains as a fast-path optimization that eliminates unnecessary INSERT attempts on clean webhook re-deliveries. The try-catch is the safety net for the concurrent-race window only.
- **Per-row catch vs `insertOrIgnore()`:** `insertOrIgnore()` bypasses Eloquent model events. The observer fires `notifyEarned()` and `notifyBrandSale()` on creation — dropping those events would suppress affiliate and brand commission notifications. Per-row catch preserves observer behavior.

## Notes

The concurrent-race scenario is inherently difficult to exercise in a synchronous unit test (would require injecting a DB write between the pre-filter and the insert). The new test covers the observable idempotency guarantee (duplicate webhook → no duplicate entries) rather than the concurrent-insert path directly. The fix itself is auditable: the catch clause is the only change to the hot path.
