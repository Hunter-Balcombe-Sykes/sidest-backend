# #3-07 — Stripe `paused` status falls through default → string literal in webhook controller

## Plain English

When Stripe tells us a subscription is "paused", the code was silently storing the raw word "paused" instead of going through the app's typed status system. This wasn't harmful in practice — the database column accepts any text — but it meant the paused status bypassed the pattern every other status uses. The fix adds `paused` as a proper named constant and routes it explicitly, so it's treated consistently with `active`, `canceled`, etc.

## Technical Summary

Two files changed:
- `app/Models/Billing/Subscription.php`: Added `STATUS_PAUSED = 'paused'` constant after `STATUS_INCOMPLETE_EXPIRED`, following the existing constant naming pattern.
- `app/Http/Controllers/Api/Webhooks/StripeWebhookController.php`: Added `'paused' => Subscription::STATUS_PAUSED` as an explicit arm in the `mapStripeStatus` match expression (line ~297), before the `default => $stripeStatus` fallback. The fallback is intentionally preserved for future unknown Stripe statuses.

No DB schema change required — `billing.subscriptions.status` is a plain `text` column with no enum constraint.

## Decisions Made

- Added `STATUS_PAUSED` to the Subscription model rather than mapping `paused` to an existing status (e.g., `STATUS_CANCELED`): a paused subscription is a distinct Stripe state (billing temporarily halted) and deserves its own constant rather than a lossy mapping.
- Preserved the `default => $stripeStatus` fallback: it remains the correct safety valve for future Stripe statuses not yet handled.

## Notes

The `billing.subscriptions.status` index and partial indexes do not reference `paused`, so no migration is needed. The `GRACE_STATUSES` array intentionally excludes `paused` — a paused subscription should not grant platform access.
