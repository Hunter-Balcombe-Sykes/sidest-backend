# #V5-022 — SendStaffBroadcastEmailsJob has no $tries or $backoff

## Plain English
The job that fans out staff broadcast emails to all subscribers had no retry logic — if something went wrong once (a blip in the database or queue), the entire broadcast would be silently dropped. We added three retry attempts with increasing wait times (10s, 30s, 60s) and a failure handler that logs the problem to Nightwatch so it doesn't go unnoticed.

## Technical Summary
Modified `app/Jobs/Notifications/SendStaffBroadcastEmailsJob` to add `public int $tries = 3`, `public array $backoff = [10, 30, 60]`, and a `failed(\Throwable $e)` method. The `failed()` handler calls `report($e)` (forwards to Nightwatch) and writes a structured `Log::error()` entry keyed by `notification_id` and `list_key`. Added `use Illuminate\Support\Facades\Log` import. Pattern mirrors the existing `SendStaffBroadcastEmailToSubscriberJob` which already had `$tries = 3`.

## Decisions Made
- `$backoff = [10, 30, 60]`: per the audit spec; the child job had no backoff, so this is a consistent upgrade.
- `failed()` body: matched the child job's pattern (`report($e)` + `Log::error`) but logged `list_key` instead of `subscription_id` since that's the parent's relevant context.

## Notes
The child job (`SendStaffBroadcastEmailToSubscriberJob`) already had `$tries = 3` but no `$backoff` — that's a separate audit item (#9-011, stage-2) and was not touched here.
