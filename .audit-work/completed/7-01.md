# #7-01 — Image MIME validation accepts client-declared Content-Type without magic-byte sniffing

## Plain English

Before this fix, when someone uploaded an image, the app trusted the label the browser put on the file — not the file itself. That meant a malicious file renamed to `.jpg` could slip past. Now we look at the actual bytes of every image before accepting it, so only real JPEGs, PNGs, and WebPs get through. Fake or disguised files are rejected with a clear error message at the moment of upload, before anything else in the system even sees them.

## Technical Summary

Created `app/Http/Requests/Concerns/SniffsFileMimeType.php` — a new trait with a single protected method `assertImageMimeBytes(UploadedFile $file, Validator $v, string $field): void`. The method reads the file's actual bytes via `(new \finfo(FILEINFO_MIME_TYPE))->file($path)` and rejects anything not in `['image/jpeg', 'image/png', 'image/webp']`. It short-circuits if the named field already has validation errors to avoid duplicate messages.

Applied the trait to three Form Request classes:
- `UploadImageRequest`: trait added; `assertImageMimeBytes` called inside the existing `withValidator` after() callback, guarded by `$hasImage`.
- `UploadBrandLogoRequest`: trait added; new `withValidator` method added checking the `logo` field.
- `UploadBrandPlaceholderImageRequest`: trait added; new `withValidator` method added checking the `image` field.

SVG is rejected implicitly — `image/svg+xml` is not in the whitelist. `UploadDocumentRequest` was not modified: the audit scope is image endpoints, and `ProfessionalDocumentController` already performs downstream finfo verification for that path.

## Decisions Made

- **Trait in `Concerns/` over a static helper class or inline code**: the codebase already has `Concerns/` for shared Form Request logic (`ValidatesProfessionalAbout`, etc.); a trait keeps the pattern consistent.
- **Skip-if-already-errored guard**: avoids presenting "not a valid image" alongside "must be a JPEG" — one error per field, first one wins.
- **UploadDocumentRequest excluded**: not named in the audit scope; the controller layer already does byte verification for that endpoint. Touching it would expand blast radius without audit justification.
- **OO finfo API (`new \finfo()`) over procedural `finfo_open/finfo_file`**: matches the style already used in `BrandDesignMediaService` and `ImageVariantService`.

## Notes

Service-layer checks in `BrandDesignMediaService` and `ImageVariantService` still exist as defense-in-depth — this fix adds the earlier gate at the HTTP boundary. The `BrandGalleryController` already used the correct `mimetypes:` Laravel rule (which does call finfo internally); this fix brings the upload request classes in line with that pattern.
