# #V5-051 — CommissionLedgerEntry FK CASCADE vs payout SET NULL — asymmetric audit retention

## Plain English
When you delete a professional's account, their commission history should be preserved for accounting and audit purposes — you need to know what commissions were earned and paid out, even after the account is gone. Previously, deleting a professional would wipe out all their commission ledger entries (the detailed record of every commission transaction). Now, deleting a professional just blanks out the reference to them in those records, but the entries themselves survive with a null where their name used to be.

## Technical Summary
Created `supabase/migrations/20260505200000_commission_ledger_entries_set_null_professional_fks.sql`.

Two changes per column (`brand_professional_id`, `affiliate_professional_id`) on `commerce.commission_ledger_entries`:
1. `ALTER COLUMN … DROP NOT NULL` — columns become nullable
2. Drop auto-generated FK constraints (`commission_ledger_entries_{column}_fkey`) and re-add with `ON DELETE SET NULL` instead of `ON DELETE CASCADE`

The `payout_id` FK was already `SET NULL` (added separately in the baseline via explicit `ADD CONSTRAINT`). This migration brings the professional FKs into alignment with that existing policy.

No PHP model, service, or test changes were required — application code always provides both professional IDs on entry creation, so the nullable columns are invisible in the production write path.

## Decisions Made
- **New migration over baseline edit**: Created a new migration file rather than retroactively editing the baseline. Supabase applies migrations sequentially; the baseline is already applied in all environments.
- **Timestamp 20260505200000**: Slots after the two existing 20260505 migrations (000000, 000001) without colliding.
- **No CHECK constraint update**: `commission_ledger_not_self_brand_check` (`affiliate_professional_id <> brand_professional_id`) evaluates to NULL when both columns are NULL, which PostgreSQL CHECKs treat as passing — no change needed.

## Notes
The `commission_payout_items` table has `commission_ledger_entry_id … ON DELETE RESTRICT`, which means payout items block ledger entry deletion. This is unaffected by our change — we're preventing professional deletion from cascading to ledger entries, not allowing ledger entry deletion.
