# #3-02 — past_due Stripe status keeps full entitlements indefinitely

## Plain English

Before this fix, if a brand's payment card failed, they kept all their plan features until Stripe eventually gave up and canceled the subscription — which could take many days. Now, brands in a "past due" state lose access to plan features after 7 days of non-payment (configurable). This gives them a reasonable window to fix their billing, but prevents indefinite free access.

## Technical Summary

**Files changed:**
- `app/Models/Billing/Subscription.php`: Rewrote `isInGracePeriod()` to replace the flat `in_array(GRACE_STATUSES)` check with status-specific logic. `STATUS_ACTIVE` always grants grace. `STATUS_PAST_DUE` grants grace only if `current_period_end` is within `sidest.billing.past_due_grace_days` days ago (default: 7). All other statuses return false. The `GRACE_STATUSES` constant is preserved unchanged as a conceptual reference.
- `config/sidest.php`: Added `billing.past_due_grace_days` key (default 7, env: `SIDEST_PAST_DUE_GRACE_DAYS`).
- `tests/Unit/Billing/SubscriptionModelTest.php`: Replaced the now-stale `'reports grace period for past_due'` test with three targeted tests: within-window → true, beyond-window → false, null period end → false.

**No Supabase migration required** — `current_period_end` already exists on `billing.subscriptions` and serves as a sufficient proxy for when the payment first failed.

## Decisions Made

- **Use `current_period_end` as proxy instead of adding `last_invoice_failed_at`**: The audit item suggested `last_invoice_failed_at` but that column doesn't exist. `current_period_end` is when the billing period ended and the invoice became due — equivalent for our purposes, and requires no DB migration.
- **Guard `config()` with `app()->bound('config')`**: Pure unit tests don't bootstrap the app container, so calling `config()` directly throws. The guard falls back to `7` in that context, matching the production default, keeping the model testable without DB/app setup.
- **Use `setRawAttributes()` in tests**: Setting a datetime-cast attribute via the normal setter (`$model->attr = value`) calls `fromDateTime()` which needs `getConnection()` → fails in unit tests. `setRawAttributes()` bypasses the cast setter; on GET, `asDateTime()` receives the Carbon instance directly and returns it without needing `getDateFormat()`.
- **Default of 7 days**: Matches the audit item's recommendation. Gives brands a realistic window for Stripe's first retry cycle without being overly generous.

## Notes

Stripe's typical dunning cycle (Smart Retries) attempts 3-4 retries over ~7 days before marking a subscription `canceled`. The 7-day default aligns with this, meaning most brands will see the subscription canceled around the same time they lose access — resulting in a clean UX rather than a "limbo" state where they have no features but also no final notice.
