## Plain English
When a broadcast email permanently fails to send to a subscriber (after all 3 retry attempts), the system now logs the failure with the notification ID, subscription ID, and error message. It also sends the error to Nightwatch so the team can observe and investigate the failure without having to dig through the raw database table.

## Technical Summary
Added `failed(\Throwable $e): void` to `app/Jobs/Notifications/SendStaffBroadcastEmailToSubscriberJob.php`. The method calls `report($e)` to surface the exception in Nightwatch, then `Log::error` with structured context (`notification_id`, `subscription_id`, `message`) for log-based querying. No interface or contract changes; this is a standard Laravel queue lifecycle hook.

## Decisions Made
- Used both `report($e)` and `Log::error`: `report()` gets the full stack trace into Nightwatch, while `Log::error` adds the structured key/value context needed to search by `notification_id` across many failures.
- Kept the log level as `error` (not `critical`): these are per-subscriber failures, not a full broadcast outage, so `error` is appropriate.
