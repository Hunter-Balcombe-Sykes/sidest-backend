# #V5-008 — Wallet currency mismatch silently fails — brand charged full amount on card

## Plain English

When a brand has wallet credits sitting in one currency (say, Euros) but their commission payout is owed in a different currency (say, Australian dollars), the system used to silently ignore the wallet and charge the brand's card the full amount. Now it detects that mismatch before any charge happens, pauses the payout, and sends the brand a notification explaining what went wrong and prompting them to contact support.

## Technical Summary

- **`app/Services/Stripe/CommissionPayoutService.php`**: Added `NotificationPublisher` injection (nullable with `app()` fallback). Modified `debitBrandManualBalancePartial()` to accept a `bool &$hasCurrencyMismatch` by-reference parameter, set to `true` when wallet balance is positive but in a different currency than the payout. In the outer transaction block, the closure captures `$hasCurrencyMismatch`; if true, the transaction returns without saving `collecting` status. After the transaction, a `wallet_currency_mismatch` check calls `markPendingFunding()` and `$this->publisher->publish()` (category `commissions`, dedupeKey `wallet_currency_mismatch.{$payout->id}`), then returns `null`. Added `formatMoney()` helper (copied from `CommissionVoidService`).
- **`tests/Feature/Stripe/CommissionPayoutServiceTest.php`**: Added two tests — one asserting the mismatch path (pending_funds, correct failure_code, wallet untouched, notification published once) and one asserting that zero-balance with a mismatched currency does NOT trigger the mismatch guard and proceeds to card charge normally.

## Decisions Made

- **by-reference bool vs exception vs separate pre-check**: Used `bool &$hasCurrencyMismatch` on the private method. Avoids creating a new exception class, avoids a second DB query, and keeps the method return type unchanged. The ref approach is minimal surface area for a private method.
- **Zero balance edge case**: Explicitly only set `$hasCurrencyMismatch = true` when `$balance > 0`. An empty wallet with any currency is "no funds", not a mismatch — prevents blocking payouts for brands whose wallets happen to have a stale non-null currency after being emptied.
- **Notification outside the transaction**: `markPendingFunding` and `publisher->publish` called after the `DB::transaction` returns. Notifications are a side effect and should not hold a DB transaction open.
- **NotificationPublisher nullable with `app()` fallback**: Matches the existing `?StripeClient` pattern in the constructor so the service remains directly constructable in tests without a container.

## Notes

- The `retryPayout` method already supports `pending_funds` status and will re-run `processPayoutBatch`, which will re-detect the mismatch on retry until the currency issue is resolved — no changes needed there.
- `formatMoney` was added as a private method (copied verbatim from `CommissionVoidService`). If this pattern repeats in more services, it's a candidate for a shared trait or helper, but three usages isn't there yet.
