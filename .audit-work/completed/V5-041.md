## Plain English

When viewing analytics, users could supply a custom start and end date and accidentally (or intentionally) request years' worth of data in a single query. The system already limited the simple "last N days" parameter to 365, but the custom date range bypassed that limit entirely. This fix adds the same 365-day ceiling to the custom from/to path, so any range wider than a year is rejected with a clear error message before any database work happens.

## Technical Summary

**Files changed:** `app/Http/Controllers/Api/Professional/ProfessionalAnalyticsController.php`

**New behavior:** After the existing `$from->gt($to)` guard in both `summary()` and `shopSummary()`, a new check `$from->diffInDays($to) > 365` returns HTTP 422 `"Date range cannot exceed 365 days."` before any DB queries are executed.

**Contracts:** The `?days` param was already capped via `max(1, min(365, $days))`. The `?from/?to` custom path now has an equivalent guard. Both methods remain in sync as required by the existing comment at line 19.

## Decisions Made

Applied the fix to both `summary()` and `shopSummary()` — the comment at the top of the controller explicitly documents that these two methods share the same date-range parsing logic and must stay in sync.
