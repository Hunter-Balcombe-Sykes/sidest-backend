# #6-02 — `public_contact_email` / `public_contact_number` echoed unconditionally in PublicSitePayload view

## Plain English

The public site view was showing a professional's contact email and phone number whenever they were set, but it wasn't obvious from the code that this was intentional. We added clear documentation explaining that these fields work as an opt-in: a professional sets them when they want that contact detail shown on their public site, and leaves them blank when they don't. The separate private email and phone (used for login and billing) are never exposed.

## Technical Summary

Created `supabase/migrations/20260505100000_document_public_contact_fields.sql` with `COMMENT ON COLUMN` statements for `core.professionals.public_contact_number` and `core.professionals.public_contact_email`, persisting the opt-in contract at the DB level (queryable via `col_description()`). Added an inline SQL comment in the `PublicSitePayload` view definition in `20260403000000_v2_baseline.sql` (near the `professional` jsonb object build) clarifying that NULL = not shared and that `primary_email`/`phone` are intentionally excluded. Updated the header comment in `app/Http/Resources/ProfessionalPublicResource.php` to note the same design contract for PHP-layer readers.

## Decisions Made

- **Documentation-only, no boolean flag**: The `public_` prefix + nullable columns already encode the opt-in contract; adding a `show_contact_info` boolean would be schema churn without new enforcement value for a P3/S item.
- **New migration rather than editing existing columns**: Column comments are schema state, so they belong in a forward migration rather than only in the baseline source file.
- **Three-layer documentation**: DB column comments, SQL view inline comment, and PHP resource comment — each targets a different reader (DBA tooling, future migration authors, API developers).

## Notes

The partial unique indexes on `public_contact_email` and `public_contact_number` (WHERE NOT NULL) already prevent two professionals from sharing the same public contact detail — a constraint that makes more sense once the opt-in design is explicit.
