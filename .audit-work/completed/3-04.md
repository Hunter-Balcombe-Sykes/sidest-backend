# #3-04 — No JSON-schema validation of webhook payload before storage

## Plain English

Before this fix, Stripe webhook payloads were stored in the database as long as the cryptographic signature checked out — but a signature can be valid over a JSON body that's missing critical fields like the event's `data` object. Now both webhook controllers check that the parsed event has an `id`, `type`, `data`, and `data.object` before saving anything to the database. If any of those are missing, the request is rejected with a 400 error and nothing is persisted.

## Technical Summary

Created `app/Http/Controllers/Api/Webhooks/ValidatesStripeWebhookPayload.php` — a trait with a single `validateEventStructure(\Stripe\Event $event): bool` method that checks `id`, `type`, `data`, and `data.object` are non-null/non-empty. Both `StripeWebhookController` and `StripeConnectWebhookController` now use this trait and call `validateEventStructure()` after `Webhook::constructEvent()` succeeds but before the `billing.webhook_events` `insertOrIgnore()`. On failure, controllers return `['error' => 'Invalid payload structure']` 400 and log a warning with field-presence details. Two new end-to-end tests were added to `StripeWebhookControllerEndToEndTest.php` and `StripeConnectWebhookControllerEndToEndTest.php`, each sending a valid HMAC over a payload missing `data` and asserting a 400 response.

## Decisions Made

- **No Opis JSON Schema library**: Installing a new package for a P3/S item is overkill. PHP property checks on the already-parsed `\Stripe\Event` object provide the same protection with zero added dependencies.
- **Trait over duplication**: Both controllers share identical validation logic; a trait in the same namespace avoids duplication without introducing a Service class (which would be over-engineering for 4 property checks).
- **Validate after constructEvent(), before insertOrIgnore()**: This is the natural seam — constructEvent() has already done signature + JSON validation, and we intercept before any side effects (DB write, event dispatch).

## Notes

The Stripe PHP SDK emits PHP `Notice: Undefined property of Stripe\Event instance: data` when our test accesses `$event->data` on an incomplete event — these notices confirmed that `constructEvent()` does NOT throw on missing fields; it silently returns null. This is the exact gap the fix closes.
