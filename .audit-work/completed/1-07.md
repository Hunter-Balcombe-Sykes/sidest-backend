# #1-07 — `EnsureSidestStaff` fail-closed but creation flow not asserted by test

## Plain English

The `EnsureSidestStaff` middleware is the door guard for internal staff routes. It requires two things: a valid Supabase login AND an entry in our staff database table. The existing tests only checked scenarios where the staff record was handed to the middleware pre-loaded — they never tested what happens when the middleware has to look the staff up in the database itself. We added two tests that exercise that database lookup: one confirms that a valid login with no matching staff record is rejected (403), and another confirms that a valid login with a matching record is allowed through.

## Technical Summary

- **File changed:** `tests/Feature/Staff/EnsureSidestStaffMiddlewareTest.php` — added two tests that leave `sidest_staff` unset on the request, forcing the middleware to execute `SidestStaff::query()->where('auth_user_id', $uid)->first()`.
- **File changed:** `tests/Pest.php` — added `setupSidestStaffTable()` helper that creates `core.sidest_staff` in the SQLite in-memory test database, following the existing `setupXTable()` pattern.
- **New test 1:** `returns 403 when supabase uid has no matching SidestStaff record in DB` — table exists but is empty; asserts 403 + `"Staff access required"` message.
- **New test 2:** `passes through when supabase uid maps to a SidestStaff record in DB` — inserts a matching row via raw DB insert; asserts 200.

## Decisions Made

- **Added helper to Pest.php rather than defining inline in the test file:** The existing convention is that DB table setup helpers are global (`Pest.php`), while test-local object factories (`makeStaffRequest`, `makeStaffInstance`) live in the test file itself. Followed that split.
- **Used raw DB insert (not Eloquent create) for the positive-path test:** Consistent with how `createTenant()`, `createCustomerFor()`, and all other test data helpers insert rows — avoids needing a factory and keeps the test independent of model event hooks.

## Notes

The existing five tests all bypass the DB lookup by pre-setting `sidest_staff` in request attributes (simulating what an upstream middleware would do in production). The new tests are the first in this file to touch the database, which is why a table setup helper was needed.
