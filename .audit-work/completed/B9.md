# B9 — `$fillable` mass-assignment cleanup

## Plain English

We found places in the code where Stripe payment IDs, deployment tokens, database keys, and audit timestamps could theoretically be overwritten through the same shortcut path we use for normal user input. That's a security hygiene problem — those fields should only be set by server code that knows what it's doing. We removed them from the "shortcut" list and updated every place in the codebase that sets those fields to use a more intentional, explicit approach. We also added tests proving those fields are now protected.

## Technical Summary

**Models changed:**
- `Subscription`: removed `stripe_customer_id`, `stripe_subscription_id` from `$fillable`
- `Plan`: removed `id` from `$fillable`
- `BrandStoreSettings`: removed `oxygen_deployment_token` from `$fillable`
- `DataExportAudit`: removed `created_at`, `completed_at` from `$fillable`; updated `markCompleted()`/`markFailed()` to set `$this->completed_at` directly before `update()`
- `ProfessionalDeletionAuditEntry`: removed `created_at` from `$fillable`; added `booted()` hook that sets `created_at = now()` on creating (matching DataExportAudit pattern)
- `CommissionPayout`, `CommissionLedgerEntry`, `BrandTeamMembership`: replaced `$fillable` array with `$guarded = ['*']`

**Callsites updated:**
- `StripeWebhookController::handleSubscriptionCreated()`: instantiates `Subscription` with mass-assignable fields then sets `stripe_customer_id`/`stripe_subscription_id` directly
- `BrandStoreSettingsController::update()`: pulls `oxygen_deployment_token` out of `$dbFields`, sets it directly after `updateOrCreate()`
- `EmbeddedSetupController::saveDeploymentToken()`: uses `updateOrCreate` for `oxygen_storefront_id`, then sets `oxygen_deployment_token` directly
- `AccountDeletionService`: removed `'created_at' => now()` from both `ProfessionalDeletionAuditEntry::create()` calls (booted hook handles it)
- `CommissionPayoutService`: `forceCreate()` for new payouts; `forceFill([])->save()` for all 9 `$payout->update()` calls and 2 `$entry->update()` calls
- `ExecuteCommissionPayoutJob`, `StripeConnectWebhookController`: `forceFill()->save()` for 3 more `$payout->update()` calls
- `ProcessShopifyOrderWebhookJob`, `ProcessShopifyOrderUpdatedWebhookJob`: `(new CommissionLedgerEntry)->forceFill($data)` instead of constructor mass assignment
- `StaffCommissionVoidControllerTest`: 5 test cases updated to `forceFill()` (companion to model change)

**Tests added:** `tests/Unit/Models/FillableHardeningTest.php` — 10 assertions covering all 8 models.

## Decisions Made

- **DataExportAudit.markCompleted/markFailed**: set `$this->completed_at` before calling `$this->update()` rather than using `forceFill` — direct property assignment marks the attribute dirty so `save()` persists it alongside the other fields in a single query.
- **ProfessionalDeletionAuditEntry booted hook**: mirrored the DataExportAudit pattern (set if not already set) rather than always overwriting, preserving backward compatibility if any caller explicitly sets created_at.
- **CommissionPayout/LedgerEntry approach**: chose `$guarded = ['*']` + `forceFill` at callsites over targeted per-field removal, per audit recommendation. All 19 callsites are server-side service/job code; no public API path reaches these models.
- **StaffCommissionVoidControllerTest**: updated constructor calls to `forceFill` as a direct companion to the model change — this is within bundle scope, not a pre-existing failure.

## Notes

- `BrandStoreSettings.clearWizardProgress()` uses `static::where()->update()` (Builder, not Eloquent instance) — this bypasses mass assignment protection and was unaffected by the `oxygen_deployment_token` removal from `$fillable`.
- `stripe_application_fee_id` was in `CommissionPayout.$fillable` but never written anywhere in the codebase — it effectively disappeared with the `$guarded = ['*']` switch.
- `default_commission_rate` (#9-004) remains in `BrandStoreSettings.$fillable` — it is user-controlled, goes through `UpdateBrandStoreSettingsRequest` validation, and is not a server-computed value. FormRequest protection is already in place.
