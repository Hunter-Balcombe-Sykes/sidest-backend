## Plain English

When a user uploads a video, the app now quickly checks whether it's actually a valid video file before saving it anywhere. If the file is corrupt, has the wrong extension, or contains no video stream, the upload is rejected immediately with a clear error message — instead of silently accepting the file and failing hours later in a background worker with a cryptic transcoding error.

## Technical Summary

**Files changed:**
- `app/Services/Media/VideoVariantService.php` — Added `probeAndValidate(string $localPath): array`. Calls the existing `probe()` method (ffprobe JSON), then asserts at least one `codec_type === 'video'` stream exists. Throws `RuntimeException` with a user-readable message if the container is unreadable or has no video stream. Returns the raw probe data so callers can reuse it if needed.
- `app/Http/Controllers/Api/Professional/Uploads/ProfessionalUploadController.php` — Imported and injected `VideoVariantService` as a third constructor parameter. Added a pre-transaction probe gate in `upload()`: if the request has a video, calls `probeAndValidate($file->getRealPath())` before creating the SiteMedia row or touching R2. On `RuntimeException`, returns 422 with `"Invalid video file: <reason>"`.
- `tests/Feature/MediaUploadFailureHandlingTest.php` — Updated the existing 503 test to mock `VideoVariantService::probeAndValidate()` returning successfully (so the test still reaches the dispatch step). Updated the existing destroy test constructor call. Added two new tests: reject when probe finds no video stream, and reject when ffprobe cannot parse the container — both assert 422 and zero SiteMedia rows created.
- `tests/Feature/Brand/BrandDesignUploadTest.php` — Fixed `makeMockedUploadController()` to pass a `VideoVariantService` mock as the new third constructor arg.
- `tests/Feature/Gallery/GalleryMixedReorderTest.php` — Fixed `callReorderController()` to pass a `VideoVariantService` mock as the new third constructor arg.

**New behaviour:** Invalid video files are rejected at the HTTP boundary (422) before any DB row or R2 object is created. Valid files proceed unchanged through the existing async transcode pipeline.

## Decisions Made

The probe runs on `$file->getRealPath()` — the PHP multipart temp file — so no extra local copy is needed. This is the cheapest point to validate: the file is already on disk, and we haven't yet touched the database or object storage.

The probe gate is positioned before the `DB::transaction` block. This means invalid files produce zero side effects: no SiteMedia row, no R2 upload, no job dispatch.
