# #8-06 — `cors.allowed_headers: ['*']` safe only because `supports_credentials: false`

## Plain English

The API config was allowing any website to send any request header, which sounds risky. It's actually safe here because the app never sends or accepts browser cookies — all auth uses JWT tokens passed explicitly. We added a comment making that dependency crystal-clear: if someone ever switches on cookie-based auth in the future, they'll see the warning right next to the wildcard settings and know they have to lock them down.

## Technical Summary

- **File changed:** `config/cors.php`
- **Change:** Replaced the one-line comment `// Wildcard headers equally safe for the same reason.` with a four-line self-contained guard that names `supports_credentials => false` as the load-bearing constraint and explicitly states that both `allowed_origins` and `allowed_headers` must be scoped to explicit values if credentials support is ever enabled.
- No behavior changes — comment only.

## Decisions Made

- **Comment verbosity:** Chose a 4-line block rather than a single-line note to match the detail level of the `allowed_origins` comment directly above. A reader skimming just the headers section should not need to look elsewhere to understand the constraint.
- **Warning phrasing:** Included "MUST be locked to explicit values" to signal a hard requirement, not a soft suggestion, since the security implication is meaningful.
