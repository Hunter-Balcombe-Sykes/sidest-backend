# #6-01 — `$clickBlockColumn` interpolated into JOIN clause in ProfessionalAnalyticsController

## Plain English
Three places in the analytics controller were building a database JOIN by pasting a variable name directly into the SQL string. While the variable could only ever be one of two known values, it looked like a potential security hole to anyone reading the code. The fix replaces each one with an explicit conditional: "if the column is block_id, join this way; otherwise, join that way." Both possible column names are now visible in the code, so the intent is clear and there's nothing that resembles dynamic SQL.

## Technical Summary
**File changed:** `app/Http/Controllers/Api/Professional/ProfessionalAnalyticsController.php`

Three `DB::table(...)->join('site.blocks as b', 'b.id', '=', "lc.{$clickBlockColumn}")` calls were replaced with `->when($clickBlockColumn === 'block_id', fn ($q) => $q->join(..., 'lc.block_id'), fn ($q) => $q->join(..., 'lc.link_block_id'))` in the closures passed to `LinkClick::runForBlockForeignKey()`.

Affected queries: `$topLinks` (line 298), `$topSections` (line 321), `$shopOpens` (line 531). No behavioral change — `runForBlockForeignKey` still tries `block_id` first and falls back to `link_block_id` on an undefined-column exception.

## Decisions Made
- Fixed all 3 occurrences, not just the two in the audit's line range: the third instance (`$shopOpens`, line 531) was identical in pattern and would have been inconsistent to leave.
- Used `when()` rather than a `match()` expression on the join string: keeps the query chain fluent and makes both full join expressions — not just the column suffix — visible to the reader.

## Notes
The `runForBlockForeignKey` resilience pattern exists to handle a column rename during schema migration (`block_id` → `link_block_id`). Once the migration is complete and the old column is dropped, all three callsites and the helper itself can be simplified to a plain join on the single surviving column.
