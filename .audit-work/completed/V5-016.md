## Plain English

When a professional uploads a document, the filename they provide is now cleaned before it gets saved — any folder-path tricks like `../../etc/passwd` are stripped down to just the filename portion, and invisible control characters (including the line-break sequences that could manipulate HTTP headers) are removed. When someone downloads the document, the filename in the browser's download prompt is now encoded using the international standard for filenames in HTTP headers, so non-English characters work correctly instead of being destroyed, and injection attacks remain impossible.

## Technical Summary

**Files changed:**
- `app/Http/Controllers/Api/Professional/ProfessionalDocumentController.php` — line 68 (`store` method): `original_filename` is now sanitized before storage using `basename()` (strips path traversal) followed by `preg_replace('/[\x00-\x1F\x7F]/', '', ...)` (strips all control chars including CR/LF). The 255-char truncation is unchanged.
- `app/Http/Controllers/Api/PublicSite/PublicDocumentDownloadController.php` — replaced the `sanitiseFilename()` allow-list strip method with `buildContentDisposition()`. The new method emits `attachment; filename="<ascii_fallback>"; filename*=UTF-8''<rfc5987_encoded>` per RFC 5987/6266. The ASCII fallback maps non-printable/non-ASCII bytes to underscores and strips `"` and `\`. The RFC 5987 value uses `preg_replace_callback` (no `u` flag, byte-by-byte) + `rawurlencode()` to percent-encode every byte outside the attr-char set — CRLF encodes as `%0D%0A`, multi-byte UTF-8 sequences encode correctly.
- `tests/Feature/Documents/DocumentControllerIntegrationTest.php` — added test `'POST /api/documents strips CRLF and path-traversal from original_filename'` asserting that `../../etc/\r\npasswd.pdf` stores as `passwd.pdf` with no CR/LF bytes.

**New behaviour:** filenames with Unicode characters are now preserved in downloads (previously destroyed by the allow-list strip); CRLF and path traversal sequences are neutralised at both storage and download time.

## Decisions Made

- Sanitization happens at storage time (not just at download time) so the stored value in `original_filename` is already clean — this protects the API response field (`buildDocumentPayload`) and any future Content-Disposition use without requiring each callsite to re-sanitize.
- The existing download controller already had a `sanitiseFilename()` call, so this was an upgrade rather than a net-new protection. The old method was kept as a replacement rather than a parallel method to avoid dead code.
- RFC 5987 encoding was chosen over a continued allow-list strip because the allow-list destroys international filenames (bad UX) while RFC 5987 handles them safely.
