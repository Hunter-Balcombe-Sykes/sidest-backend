# #2-04 — Theoretical race in ProfessionalCacheService on `auth_user_id` change mid-request

## Plain English

The audit flagged a potential race condition where a user's identity could theoretically change partway through a request. In practice this can't happen — once a professional account is created, the auth ID tied to it never changes. We added a short comment to the code to explain this assumption, so future developers don't waste time worrying about the race or, worse, adding unnecessary locking around it.

## Technical Summary

Added a docblock to `ProfessionalCacheService::getByAuthId()` (`app/Services/Cache/ProfessionalCacheService.php`) documenting that `auth_user_id` is immutable (set at account creation, never updated). The comment clarifies that the existing mismatch guard (lines 142–157) is a stale/corrupt-cache defence, not a concurrency fix, and that no real mid-request race exists. No logic changes were made.

## Decisions Made

- **Comment form — docblock not inline**: The note applies to the whole method contract, not a single block, so a docblock above the method signature fits CLAUDE.md's "brief docblocks on public service methods" guidance better than an inline comment.
- **No code changes**: The existing defensive guard is correct as-is; documenting the assumption is the entire fix.

## Notes

The mismatch guard is genuinely useful as a stale-cache safety net even if the race is theoretical — deliberately left untouched.
