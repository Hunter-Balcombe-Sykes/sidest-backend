# #V5-007 — Transfer idempotency key changes on retry — duplicate transfer risk

## Plain English

When a Stripe transfer to an affiliate succeeds on Stripe's end but the HTTP connection drops before Side St can record it, Horizon automatically retries — safely, because it uses the same idempotency key. The bug was that *admin* manual retries (via the `retryPayout()` method) increment a retry counter, which was being appended to the transfer's Stripe idempotency key. So a second admin retry after a network outage would send a fresh key, and Stripe — not recognising it — would create a second transfer, paying the affiliate twice. The fix makes the transfer key permanent (no retry suffix), so Stripe always returns the same transfer regardless of how many times an admin retries.

## Technical Summary

**File changed:** `app/Services/Stripe/CommissionPayoutService.php`

Three coordinated changes in `processPayoutBatch()`:

1. **Stable transfer idempotency key** — changed from `'tr_'.$payout->id.$retryKey` to `'tr_'.$payout->id`. The `$retryKey` suffix (containing `retry_count`) is now only applied to PaymentIntent keys, where it is still needed to bypass Stripe's 24h error-cache after terminal card failures.

2. **`stripe_transfer_id` guard** — added `if (! $payout->stripe_transfer_id)` before the transfer creation block. Skips the Stripe call and jumps directly to marking the payout `completed` if the transfer ID was already recorded. Guards against a crash between the two saves introduced below.

3. **Split save** — separated the single `forceFill([stripe_transfer_id, status=completed, ...])->save()` into two sequential saves: first `stripe_transfer_id`, then the completion fields. This makes `stripe_transfer_id` a durable checkpoint that survives a process crash between the two DB writes.

**Test file changed:** `tests/Feature/Stripe/CommissionPayoutServiceTest.php` — added `'skips transfer creation and completes when stripe_transfer_id was already recorded'` to cover the new guard; updated the `retry_count` test description to accurately reflect that only the PaymentIntent key is refreshed on admin retry, not the transfer key.

## Decisions Made

- **Stable key for transfer only, not PaymentIntent**: The PI key still uses `retry_count` because terminal card failures are common and you need a fresh key to bypass the 24h Stripe error-cache on admin retry. Transfer terminal failures are rare, and the 24h window is acceptable given that the underlying cause (invalid connected account, etc.) rarely resolves faster anyway.
- **Split save over single atomic save**: A two-step save is slightly less atomic, but the `stripe_transfer_id` guard makes it safe: if the process crashes after saving the transfer ID but before saving `status=completed`, the next retry detects the ID and finalises without re-calling Stripe.
- **Guard added even though currently unreachable**: `stripe_transfer_id` can only be set if `status=completed` in the old single-save path (making the guard logically redundant). With the split save introduced here, the guard is now actively load-bearing for crash recovery.

## Notes

The root trigger for this scenario: `ExecuteCommissionPayoutJob::failed()` sets `status=failed` when all Horizon retries are exhausted (including during the transfer step), which opens the payout to `retryPayout()`, which increments `retry_count`. With the old code this created a window for duplicate transfers. The stable key closes that window entirely — Stripe's idempotency layer becomes the de-facto checkpoint regardless of how many admin retry cycles occur.
