## Plain English

The list of valid image pools (`gallery` and `content`) was copy-pasted into two separate upload validation files, which meant updating it in the future required finding and changing two places. Now that list lives in one place — the main config file — and both validation files read from it automatically.

## Technical Summary

**Files changed:**
- `config/sidest.php` — added `upload_pools` key (`['gallery', 'content']`) alongside the existing `image_pools` block, with a comment explaining it governs the generic professional upload endpoint specifically
- `app/Http/Requests/Api/Professional/Uploads/UploadImageRequest.php` — replaced `Rule::in(['gallery', 'content'])` with `Rule::in(config('sidest.upload_pools'))`; also replaced the hardcoded error message `'Pool must be "gallery" or "content".'` with a dynamic `implode()`-based version
- `app/Http/Requests/Api/Professional/Uploads/ReorderPoolImagesRequest.php` — same `Rule::in` replacement

**Behavior:** Unchanged — `gallery` and `content` remain the only accepted values.

## Decisions Made

Used a new `upload_pools` key rather than `array_keys(config('sidest.image_pools'))` because `image_pools` contains 6 pools (including `product`, `brand_gallery`, `product_custom`, `documents`) — each with dedicated controllers that carry their own authorization logic. Expanding the generic endpoint to all pools would silently bypass that pool-specific auth. The named `upload_pools` key keeps intent explicit and the restriction intact.
