# #5-02 — Square/Fresha disconnect doesn't revoke tokens at provider

## Plain English

When a professional disconnects Square or Fresha, we now ask the provider to invalidate the OAuth token before we delete our local copy. If the provider refuses or is unreachable we log a warning and proceed with the disconnect anyway — the user always gets cleanly disconnected on our end, but the token at the provider is now also killed in the happy path.

## Technical Summary

- **`app/Services/Square/SquareTokenService::revokeToken(ProfessionalIntegration)`** — new method. Posts to Square's `/oauth2/revoke` endpoint with `Authorization: Client {client_secret}` header (Square's required scheme for revocation) and the access token in the JSON body. Returns void; does not throw.
- **`app/Services/Fresha/FreshaTokenService::revokeToken(ProfessionalIntegration)`** — new method. Posts to Fresha's `/oauth2/revoke` endpoint following RFC 7009 conventions (credentials + `token` field in body). Returns void; does not throw.
- **`SquareIntegrationController::disconnect()`** — injected `SquareTokenService` via constructor. Fetches the integration model before deletion, calls `revokeToken()` in a `try/catch(\Throwable)`, logs a `warning` on failure, then proceeds to delete the local row.
- **`FreshaIntegrationController::disconnect()`** — identical pattern with `FreshaTokenService`.

The `disconnect()` methods previously used a query-builder `->delete()` without loading the model; they now load the integration first to obtain the access token for revocation.

## Decisions Made

- **Log-and-continue on failure**: Wrapped revoke in `try/catch(\Throwable)` with `Log::warning()`. Revoke failure must not prevent disconnection — a dangling remote token is a hygiene issue, not a showstopper.
- **Square auth scheme**: Used `Authorization: Client {client_secret}` header as required by Square's revoke endpoint — distinct from the bearer-token pattern used on all other Square API calls. This is not obvious from the service's existing code.
- **Fresha endpoint is inferred**: Fresha's Partner API has no publicly confirmed revoke endpoint. Used `/oauth2/revoke` per RFC 7009 conventions and left the existing NOTE comment in `FreshaTokenService`. The call will simply fail silently (caught and logged) if the endpoint doesn't exist.
- **Constructor injection over `app()`**: Added `__construct` to both controllers rather than resolving services inline, matching standard Laravel DI conventions.

## Notes

The Fresha revoke call is best-effort. If Fresha does expose a revoke endpoint its URL and payload shape should be confirmed against Partner API docs and the NOTE in `FreshaTokenService::revokeToken()` updated accordingly.
