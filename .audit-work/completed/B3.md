# B3 — Auth-surface mechanical cleanup

## Plain English

Any professional (affiliate, influencer) could previously reach the invite creation and deletion endpoints — the only gate was a payment check that was explicitly designed to wave non-brands through. We added a role check (`brand.only`) as the outer gate so those routes now reject non-brands before the payment check even runs. The unauthenticated Hydrogen affiliate endpoint got a clear "intentionally unlocked" label so future engineers know the security contract depends on the controller's link verification, not route middleware. A duplicate key-registration line in the security policy boot was removed — it was a copy-paste error, not a missing registration for another model.

## Technical Summary

**routes/api/professional.php:** The three `POST /brand-affiliate-invites` routes and the `DELETE /brand-affiliate-invites/{invite}` route are now wrapped in `Route::middleware(['brand.only'])`. `brand-funding-gate` is nested inside the `brand.only` group, preserving the payment verification for brands while ensuring non-brands are rejected at the role gate first. Updated the comment to clarify both gates and their distinct purposes.

**routes/api.php:** Replaced the prose comment above `Route::get('/internal/hydrogen/affiliate')` with an explicit `INTENTIONALLY UNAUTHENTICATED` annotation that names the specific security contract: `HydrogenAffiliateController::show()` enforces a 404 for any input that doesn't resolve to a verified `BrandPartnerLink`. Confirmed the controller does enforce this at three levels (shop_domain, affiliate slug, BrandPartnerLink existence).

**app/Providers/AppServiceProvider.php:** Removed the duplicate `Gate::policy(\App\Models\Retail\CommissionMovement::class, \App\Policies\CommissionPolicy::class)` line (was on consecutive lines 46–47). Confirmed no `CommissionLedgerEntry` model exists in the codebase — no missing registration was masked by the duplicate.

## Decisions Made

- **brand.only wraps brand-funding-gate, not the reverse**: The audit item specified this ordering. `BrandFundingGate` explicitly passes non-brands through (by design), so it must be the inner gate. Outer `brand.only` short-circuits with 403 before payment verification runs for non-brands.
- **DELETE route moved inside the brand.only group**: The audit noted DELETE had zero role protection. It does not need `brand-funding-gate` (deletion doesn't involve payment float), so it was placed inside `brand.only` but outside the funding gate group.
- **AUTH-3: comment-only fix, no controller changes**: The controller already enforces 404 at three levels. The audit's "if not, add hydrogen.key" branch did not apply.
- **AUTH-5: straight deletion, no replacement**: Glob confirmed no `CommissionLedgerEntry` model exists — the duplicate was not masking a missing registration for a renamed model.

## Notes

The `BrandFundingGate` comment in the middleware source (`Non-brand requests pass through. The route's own role check... is the right place to 403`) was written anticipating that a `brand.only` gate would eventually be added here. AUTH-2 closes that acknowledged gap.
