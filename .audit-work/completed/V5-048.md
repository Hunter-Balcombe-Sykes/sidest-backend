## Plain English

The staff access system now has formal role definitions that prevent accidental escalation in code. Previously, the word "admin" was scattered as a raw string in multiple files, making it easy to mistype or forget a check. Now there are named constants (`ROLE_SUPPORT`, `ROLE_ADMIN`) and a dedicated `isAdmin()` method, so the role check is always done the same way. Any route can now declare `middleware('staff:admin')` directly — no need for a separate admin middleware alias — which makes it easier to add new fine-grained roles in the future.

## Technical Summary

**Files changed:**

- `app/Models/Core/Staff/SidestStaff.php` — Added `ROLE_SUPPORT = 'support'` and `ROLE_ADMIN = 'admin'` constants plus an `isAdmin(): bool` helper method.
- `app/Http/Middleware/Auth/EnsureSidestStaff.php` — Added variadic `string ...$roles` parameter to `handle()`. When roles are passed (e.g., `middleware('staff:admin')`), the middleware enforces the role check after authenticating the staff record. Also updated to check request attributes first before querying the DB (eliminates double lookup on admin routes where both `staff` and `staff.admin` run).
- `app/Http/Middleware/Auth/EnsureSidestAdmin.php` — Replaced `$staff->role !== 'admin'` magic string with `! $staff->isAdmin()`. No behavior change.
- `app/Http/Controllers/Api/Staff/ProfessionalSiteManagement/StaffDataExportController.php` — Replaced inline `$staff->role !== 'admin'` with `! $staff->isAdmin()`. No behavior change.
- `tests/Feature/Staff/EnsureSidestStaffMiddlewareTest.php` — New: 5 tests covering no-restriction passthrough, role match, role mismatch (403), missing UID (401), and multi-role acceptance.

**New contract:** `middleware('staff:admin')` is now equivalent to the existing `staff.admin` alias; both remain valid. The `staff.admin` alias (EnsureSidestAdmin) is preserved for backward compatibility.

## Decisions Made

The `EnsureSidestAdmin` middleware was kept intact rather than removed. Routes in `routes/api/staff.php` already use `staff.admin` and changing them would widen the blast radius unnecessarily. The parameterized approach (`staff:admin`) is additive — new routes can use it, old routes keep working.

The "attributes-first" DB lookup optimization was included because the `EnsureSidestAdmin` middleware already did this pattern, so aligning `EnsureSidestStaff` was a natural consistency fix.
