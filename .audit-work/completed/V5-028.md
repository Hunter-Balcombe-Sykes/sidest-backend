## Plain English

The `laravel/tinker` package (a developer REPL — basically a live PHP console for the app) was accidentally listed as a production dependency, meaning it gets installed on every server deployment. It has no use in production and gives an attacker a convenient execution path if the server is ever compromised. Moving it to the dev-only list ensures it never ships in production.

## Technical Summary

**Files changed:** `composer.json`, `composer.lock`

**What changed:** Moved `laravel/tinker` from the `require` block to `require-dev` in `composer.json`. Ran `composer update laravel/tinker` to regenerate the lockfile; Composer also upgraded tinker from v2.10.2 to v2.11.1 during that run.

**New behavior:** Any `composer install --no-dev` (standard for production deployments) will now exclude tinker entirely. Dev environments are unaffected.

**Contracts unchanged:** No application code references tinker directly; no service providers, no routes, no tests touch it. The `package:discover` step re-ran cleanly and tinker's service provider is still auto-discovered for local dev.

## Decisions Made

No questions arose — the fix was unambiguous. The minor version bump (2.10.2 → 2.11.1) was accepted since it was pulled in automatically and tinker is dev-only; no risk to production.
