# #CR-014 — Shop block_id can be null with state:'live'; Hydrogen tracking pipeline silently drops null block_ids

## Plain English

When a new affiliate hasn't yet set up their shop section, the Side St backend was sending the frontend a tracking ID of "null" for that section. The frontend's analytics code silently ignores null IDs — so zero shop clicks were recorded, with no error or warning to indicate something was wrong. The fix changes the backend to simply not include the tracking ID field at all when the shop block doesn't exist. The frontend can now distinguish "no block configured" (field absent) from "block configured but unpublished" (field present).

## Technical Summary

**File changed:** `app/Http/Controllers/Api/Internal/HydrogenAffiliateController.php`

`sectionEnvelope()` (line 172) previously returned `block_id: $section?->id` unconditionally, emitting `block_id: null` for any section type whose Block row hasn't been created yet. The fix restructures the return to build `$envelope` without `block_id`, then conditionally sets `$envelope['block_id'] = (string) $section->id` only when `$section !== null`. The `@return` PHPDoc is updated to `array{state: string, block_id?: string, data: mixed}` to reflect the optional key.

The contract change applies to all section types (gallery, bio, newsletter, services, booking, shop) — not shop alone — since all share the helper. Any Hydrogen consumer firing analytics on `block_id` gains the same guard-on-presence guarantee.

**File added:** `tests/Feature/Api/HydrogenAffiliateSectionEnvelopeTest.php` — three Pest unit tests exercising `sectionEnvelope()` via `ReflectionMethod`: (1) no block → key absent, (2) block live → key present with UUID, (3) block draft → key present with UUID but state=draft.

## Decisions Made

- **Applied fix to all sections via shared helper, not shop-only**: Null block_ids are silently dropped by Hydrogen's pipeline for any section type, not just shop. Fixing the helper universally is more consistent and avoids a special-case branch.
- **Used key absence rather than null**: `block_id: null` is ambiguous ("API returned null" vs "field missing"). Key absence is unambiguous — Hydrogen can use `'block_id' in shop` as the guard condition.
- **Used ReflectionMethod for testing private method**: The private method is the correct unit to test here. Making it `protected` just for testability would have been a larger visibility change with no other benefit.
- **Cast id to string explicitly**: `(string) $section->id` ensures a consistent wire type even if the model's UUID is stored as a non-string internally.

## Notes

The only place `block_id` was documented as purposeful (the shop section comment at line 102-104) was updated to document the new contract ("block_id is absent, not null — Hydrogen guards on presence"). Other sections that also use sectionEnvelope (gallery, bio, etc.) gain the same contract silently — no comment needed there since the pattern is consistent.
