## Plain English

When a commission payout fails to transfer to an affiliate and the automatic refund to the brand also fails, the brand could end up charged with no way for the team to notice. We've added a flag that gets set in this exact scenario, and staff can now filter for these payouts specifically in the dashboard. Before retrying the payout, staff must go into Stripe and confirm the refund was issued, then call a new acknowledgment endpoint — this prevents accidentally double-charging the brand.

## Technical Summary

**Files changed:**
- `supabase/migrations/20260504000000_add_needs_manual_refund_to_commission_payouts.sql` — adds `needs_manual_refund BOOLEAN NOT NULL DEFAULT FALSE` column + partial index to `commerce.commission_payouts`
- `app/Models/Retail/CommissionPayout.php` — adds `needs_manual_refund` boolean cast
- `app/Services/Stripe/CommissionPayoutService.php` — sets `needs_manual_refund = true` via `forceFill` in the `catch (\Throwable $refundEx)` block (line ~491); clears it to `false` in `retryPayout()` reset block
- `app/Http/Controllers/Api/Staff/ProfessionalSiteManagement/StaffCommissionPayoutController.php` — three changes: (1) `index()` accepts `?needs_manual_refund=true/false` filter, (2) `retry()` returns explicit 422 when `needs_manual_refund=true` with a descriptive message, (3) new `acknowledgeManualRefund()` method that clears the flag, clears `stripe_payment_intent_id`, and resets `failure_code` to `transfer_failed` (making the payout retryable)
- `routes/api/staff.php` — registers `POST /staff/commission-payouts/{payout}/acknowledge-manual-refund` in the admin route group
- `tests/Feature/Stripe/CommissionPayoutServiceTest.php` + `ExecuteCommissionPayoutJobTest.php` — added `needs_manual_refund INTEGER NOT NULL DEFAULT 0` to inline SQLite test schemas

**New behavior:**
- Double-failure scenario (transfer fails + auto-refund fails): `needs_manual_refund = true` is persisted on the payout record
- Staff can filter: `GET /staff/commission-payouts?needs_manual_refund=true`
- Retry is blocked (422) until staff calls: `POST /staff/commission-payouts/{payout}/acknowledge-manual-refund`
- Acknowledge clears the flag, wipes the PI ID (avoids Stripe's 24h idempotency TTL returning the refunded PI), and makes the payout retryable via the existing `/retry` endpoint

## Decisions Made

- `acknowledge` changes `failure_code` from `transfer_failed_refund_needed` to `transfer_failed` rather than null — preserves the payout history showing it went through a refund scenario while making it retryable (the `retryPayout` guard checks for the specific `transfer_failed_refund_needed` code, not for `transfer_failed`)
- The `needs_manual_refund` flag is redundant with `failure_code = transfer_failed_refund_needed` but intentionally separate: booleans are indexable, filterable, and convey intent more clearly to the staff API consumer than a string code
