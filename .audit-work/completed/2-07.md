# #2-07 — Cache keys don't include site_id (one-site-per-pro assumption)

## Plain English

Right now every professional has exactly one website, so caching analytics data under the professional's ID works fine. But if we ever let a professional own multiple sites, those cache entries would get mixed up — site A's visitor counts would bleed into site B's dashboard. We added clear notes in the code spelling out this assumption and flagging exactly which cache keys would need to be updated if multi-site support ever ships.

## Technical Summary

`app/Services/Cache/CacheKeyGenerator.php` received a class-level doc comment explaining the one-site-per-professional assumption: keys that cache site-scoped data (analytics, commerce) are currently namespaced by `professionalId` rather than `siteId`. Six methods were annotated with `@multi-site: needs site_id` inline comments: `analyticsVisits`, `analyticsClicks`, `analyticsSummary`, `analyticsSummaryVersion`, `bookingAnalytics`, `affiliateCommerceAnalytics`, and `brandCommerceAnalytics`. No key schemas or method signatures were changed.

## Decisions Made

- **Document only, no key restructuring**: Changing key signatures would require updating all callers and cache-busting existing entries. At P3/S effort the audit explicitly offered documentation as a valid resolution, and it achieves the goal without a breaking change.
- **`@multi-site` annotation style**: Chose a terse inline comment tag rather than full docblocks — keeps the annotation scannable and grep-able without adding prose noise.
- **Which methods to annotate**: Restricted annotations to methods where the cached value is logically site-scoped (visit/click/summary analytics, commerce analytics). Left `professionalServices`, `customerCount`, and brand-catalog keys unannotated because those are pro-level rather than site-level data and wouldn't need a site_id segment even under multi-site.

## Notes

The methods that are genuinely safe (services, customer count, brand catalog) are keyed by professional for the right reason — they don't aggregate traffic or revenue per-site. Only the analytics pipeline would need restructuring under multi-site.
