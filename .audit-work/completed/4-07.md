## Plain English

When a brand connects their Shopify store, five background jobs run to finish setting up the integration. Previously, if any of those jobs failed permanently, there was no way to know which ones broke or kick off a recovery. Now each job records its own success or failure in the integration record, the dashboard API exposes a clear "setup incomplete" status, and there's a new retry endpoint so a brand can re-run only the failing steps without reinstalling the whole app.

## Technical Summary

**Files changed:**

- `app/Jobs/Shopify/CreateStorefrontAccessTokenJob.php` — added `storefront_token_state: 'registered'` on success (including early-return idempotent path) and `'failed'` in `failed()`
- `app/Jobs/Shopify/SyncShopifyBrandDesignJob.php` — added `brand_design_state: 'synced'` on success and `'failed'` in `failed()`
- `app/Models/Core/Professional/ProfessionalIntegration.php` — added `shopifyInstallStatus(): array` which reads the 5 per-job state keys (`webhooks_state`, `metafield_definitions_state`, `sales_channel_state`, `storefront_token_state`, `brand_design_state`) from `provider_metadata` and returns an umbrella `state` (`complete` | `incomplete` | `pending`) plus per-step detail
- `app/Http/Controllers/Api/Professional/ShopifyIntegration/ShopifyIntegrationController.php` — `status()` now includes `setup_state` and `setup_steps` fields; added `retrySetup()` action that re-dispatches only non-success steps after resetting their state to `'queued'`
- `routes/api/professional.php` — added `POST /shopify/setup/retry`
- `tests/Feature/Shopify/ShopifySetupStateTest.php` — 8 new tests covering all state combinations and controller behaviour

**New behaviour:**
- `GET /shopify/status` returns `setup_state: 'complete'|'incomplete'|'pending'` and per-step breakdown when a token is provisioned
- `POST /shopify/setup/retry` re-dispatches failing/pending steps; returns immediately (async). `ShouldBeUnique` on each job prevents duplicate concurrent runs. Callers poll `/shopify/status` for progress.
- No DB schema changes needed — state lives in the existing `provider_metadata` jsonb column.

## Decisions Made

- **`'incomplete'` vs `'pending'`**: `incomplete` means at least one job permanently failed after all retries; `pending` means jobs are still in-flight or queued. This distinction lets the frontend show "Setup incomplete — Retry" only when it's definitely broken, not while jobs are still running.
- **Retry resets state to `'queued'`** rather than null, so `shopifyInstallStatus()` always returns `pending` (not `incomplete`) immediately after a retry call — avoids the frontend flickering between "incomplete" and "pending".
- **No state column added** — the existing `provider_metadata` jsonb is sufficient. A dedicated column would require a Supabase migration with no material benefit for a single-provider state machine of this complexity.
