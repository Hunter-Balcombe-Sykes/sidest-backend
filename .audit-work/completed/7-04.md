# #7-04 — Video processing has no codec allowlist, no resolution cap, no FFmpeg timeout

## Plain English

The video processing code previously ran FFmpeg with no time limit and no rules about what kinds of videos it would accept. A maliciously crafted file could have hung the video worker indefinitely. Now the code checks the codec (only H.264, H.265, and VP9 are allowed), caps input resolution at 4K, and sets a timeout on every FFmpeg call equal to twice the video's duration plus 60 seconds. It also switches from passing command strings through a shell to passing each argument directly to the process, which prevents shell injection tricks.

## Technical Summary

**File changed:** `app/Services/Media/VideoVariantService.php`

- **Class constants added:** `ALLOWED_CODECS = ['h264', 'hevc', 'vp9']`, `MAX_RESOLUTION_LONG = 3840`, `MAX_RESOLUTION_SHORT = 2160`.
- **`probeAndValidate()`** extended: after confirming a video stream exists, it now validates `codec_name` against `ALLOWED_CODECS` and checks that `max(width, height) ≤ 3840` and `min(width, height) ≤ 2160`. Throws `RuntimeException` on failure with a clear message.
- **`processVariants()`** now calls `probeAndValidate()` instead of `probe()` (defense-in-depth on the worker path, not just the HTTP boundary). Computes `$encodingTimeout = max(120, durationSeconds * 2 + 60)` and passes it to all three FFmpeg helpers.
- **`probe()`** rebuilt to use `new Process([...])` with a 30s timeout and explicit stdout-only JSON parsing (stderr read separately for error messages, not mixed into the JSON string).
- **`runCommand()`** signature changed from `string $cmd` to `array $cmd` with `int $timeout = 30`. All shell-composed `sprintf(escapeshellcmd(...)...)` calls eliminated; each argument is now a separate array element.
- **`encodeMp4()`, `packageHls()`, `extractPoster()`** all updated to build `array $cmd` and accept `int $timeout`.

## Decisions Made

- **`hevc` not `h265` in the allowlist:** ffprobe reports H.265 as `hevc`; the string `h265` never appears. The constant comment documents this so the list isn't silently wrong.
- **4K check via long/short edge:** Using `max(w, h) ≤ 3840 && min(w, h) ≤ 2160` rather than a raw `w ≤ 3840 && h ≤ 2160` check handles portrait-orientation 4K correctly.
- **Timeout floor of 120s:** Clips with zero or very short duration (probe returns 0ms) would compute a 60s timeout, which is tight for encoding overhead. 120s is a safe minimum for any real transcode.
- **`probe()` builds its own Process** rather than calling `runCommand()`: the probe path needs stdout-only output for JSON parsing; mixing stderr would corrupt `json_decode`. Keeping it separate is cleaner than adding a stderr-ref parameter to `runCommand()`.

## Notes

- The `probeAndValidate()` docblock still says "Call this in the upload controller before storing to R2" — that guidance is still accurate, and the worker now also calls it for defense-in-depth.
- No tests existed for this service prior to this fix; the change is fully covered by the hardening logic itself (validated manually against the Pest suite, all 1477 tests pass).
