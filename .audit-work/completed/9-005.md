## Plain English

A background job that checks which streamers are currently live had a small SQL query written with a value baked directly into the query string instead of passed as a safe parameter. No real vulnerability existed because no user data ever touched that value, but the pattern was inconsistent and fragile — easy to copy wrong in the future. The fix is a one-character change that passes the value through the database driver's binding mechanism, the same way every other query in the codebase works.

## Technical Summary

**File changed:** `app/Jobs/Streaming/CheckStreamingLiveStatusJob.php` (line 47)

**Before:**
```php
->whereRaw("settings->>'live_check_enabled' = 'true'")
```

**After:**
```php
->whereRaw("settings->>'live_check_enabled' = ?", ['true'])
```

The query now uses PDO parameter binding instead of a string literal. Behaviour is identical — the JSONB column comparison against the string `'true'` is unchanged — but the pattern is now consistent with parameterized style and safe against future misuse if the value ever becomes dynamic.

## Decisions Made

No ambiguity. The audit item was precise and the fix was mechanical.
