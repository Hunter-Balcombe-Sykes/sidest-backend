# #7-09 — VideoVariantService uses `Process::fromShellCommandline` (currently safe, server-config inputs only)

## Plain English

The audit flagged that video processing code might be building shell commands as a single string, which could be dangerous. When the code was checked, it was already using the safe form everywhere — each command-line argument is passed as a separate item in a list rather than being combined into one string. A short explanatory comment was added to the one spot that lacked it, so both places now document why they use this approach.

## Technical Summary

`VideoVariantService.php` has two `Process` call sites:
- `probe()` (line 64): direct `new Process([...])` — already array-form
- `runCommand()` (line 542): `new Process($cmd)` where `$cmd` is always an array — already array-form

No `Process::fromShellCommandline` usage existed anywhere in `app/`. The fix was purely documentary: added a one-line inline comment above `probe()`'s `new Process` instantiation noting that array-form is intentional and bypasses shell interpolation, matching the rationale already in `runCommand()`'s docblock.

## Decisions Made

- Chose a documentation-only commit rather than refactoring `probe()` to route through `runCommand()`: `probe()` needs stdout and stderr separated (stdout for JSON parsing, stderr for the error message), while `runCommand()` combines them. Routing through `runCommand()` would require a signature change to `runCommand()` that is out of scope.
- Used an inline comment rather than expanding the `probe()` docblock: the WHY is a single sentence and doesn't warrant a full docblock addition.

## Notes

The audit item was generated before the #7-04 hardening was applied; #7-04 introduced `runCommand()` with the array-form pattern and added the docblock explaining it. The `probe()` method predates `runCommand()` and retained a direct `new Process` call that was already array-form but lacked the explanatory comment.
