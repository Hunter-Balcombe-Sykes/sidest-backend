## Plain English

When the app builds the HLS video playlist file, it was plugging values from the config directly into the file without checking them first. If a config key or resolution value ever contained unexpected characters (like a line break), it would quietly break the playlist format. The fix adds a quick check — if those values don't look exactly right, the job fails loudly instead of producing a corrupted file.

## Technical Summary

**File changed:** `app/Services/Media/VideoVariantService.php` — `buildAdaptivePlaylist()` method (~line 435)

**Before:** `$variantKey` (config array key) and `$resolution` (config string value) were interpolated into M3U8 playlist lines without any validation, allowing newlines or special chars to corrupt the line-based HLS format.

**After:** Two `preg_match` guards added before interpolation:
- `$variantKey` must match `/^[a-zA-Z0-9_-]+$/` (alphanumeric, dashes, underscores only)
- `$resolution` must match `/^\d+X\d+$/` after `strtoupper()` normalisation (e.g. `1280X720`)

Both throw `\RuntimeException` on failure, consistent with existing config-validation patterns in this service (see duration check at line 124).

## Decisions Made

Regex validated *after* `strtoupper()` so a single pattern covers both `1280x720` and `1280X720` from config without needing a case-insensitive flag.
