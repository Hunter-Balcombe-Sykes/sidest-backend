# B2 — Media job terminal-state idempotency guards

## Plain English

After a video or image upload is processed by a background worker, the system now correctly stamps it as "done" so the dashboard stops showing a loading spinner. Previously, videos were processed successfully by FFmpeg but the database record never got updated to say "ready," so every video appeared permanently stuck. Additionally, both video and image processing jobs now check at the start whether the work was already done — so if a worker restarts and accidentally picks up a completed job again, it skips cleanly instead of reprocessing and potentially marking a perfectly fine asset as broken.

## Technical Summary

**Files changed:** `app/Jobs/ProcessVideoVariantsJob.php`, `app/Jobs/ProcessImageVariantsJob.php`

**ProcessVideoVariantsJob (JOB-1):**
- Added `processing_state = PROCESSING_STATE_READY` write (with `processing_error = null`) immediately after `VideoVariantService::processVariants()` returns on the happy path. This completes the `pending → processing → ready` state machine that was documented in the class docblock but only partially implemented (the `→ failed` branch via `markFailed()` was already correct).
- Added idempotency guard after the trashed-row check: if `processing_state` is already `PROCESSING_STATE_READY` or `PROCESSING_STATE_FAILED`, the job logs and returns early. Uses the `$siteMedia` instance already loaded at job entry — no extra query.

**ProcessImageVariantsJob (JOB-2):**
- Added identical idempotency guard (same placement, same logic) to prevent redelivered jobs from overwriting a `ready` row back to `processing` and then potentially landing it in `failed` via `markFailed()` + `cleanupR2Artifacts()`.

Both guards use `in_array(..., true)` (strict) and reference the class constants. The ready-state write in the video job mirrors lines 110–116 of the image job exactly.

## Decisions Made

- **Log line on idempotency skip:** Added `Log::info(...)` matching the pattern of existing skip paths (missing row, soft-deleted row). This gives observability into how often redeliveries occur in production.
- **No cache bust in video job:** The image job busts `SiteCacheService::forgetBrandDesign` on `POOL_DESIGN` rows. The audit item specified mirroring only the DB update block (image job lines 109–115), not the cache bust. Videos in `POOL_DESIGN` are not an established use case in the current codebase, so no cache bust was added to the video job.
- **Placement before the `processing` write:** The idempotency guard is inserted between the trashed-row check and the `SiteMedia::query()->update(['processing_state' => PROCESSING_STATE_PROCESSING, ...])` line, so a terminal row is never overwritten back to `processing`.

## Notes

The video job's `failed()` hook and `markFailed()` path were already correct — only the success path was broken. The three-state machine was fully documented in the docblock, which made the gap easy to spot: the docblock listed `processing → ready` as a transition but the code had no write implementing it.
