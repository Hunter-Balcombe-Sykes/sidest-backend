# B14 — Throttle config hardening (P2 only)

## Plain English

Four health-check endpoints (`/ping`, `/health`, `/ready`, `/health/scheduler`) were completely open and could be hammered without limit — we added a 60-requests-per-minute cap per IP to all of them. We also added a safety check so the app refuses to start in production if someone accidentally turns off all rate limiting via an environment variable. Finally, we made the system loudly crash (instead of silently degrading) if a programming mistake removes the required login middleware from authenticated routes, which would have caused per-user rate limits to accidentally merge everyone behind the same IP.

## Technical Summary

**Files changed:** `routes/api.php`, `app/Providers/AppServiceProvider.php`

**V5-057:** Registered a new `health-check` named rate limiter in `AppServiceProvider::configureRateLimiting()` (60/min by IP, respects `$throttleEnabled`). Applied `->middleware('throttle:health-check')` to `/ping`, `/health`, `/ready`, and `/health/scheduler` routes.

**V5-059:** Added a guard in `AppServiceProvider::boot()` (before `configureRateLimiting()` is called) that throws `\RuntimeException` if `app()->isProduction()` is true and `config('sidest.throttle.enabled')` is false. Prevents misconfigured `SIDEST_THROTTLE_ENABLED=false` from reaching production.

**V5-060:** Replaced the `$request->attributes->get('supabase_uid') ?? $request->ip()` pattern with a null-coalescing throw in five rate limiters: `affiliate-writes`, `brand-catalog-writes`, `authenticated`, `staff`, and `bootstrap`. Each throws `\RuntimeException` with a descriptive message naming the limiter if `supabase_uid` is absent, producing a 500 that surfaces in Nightwatch rather than silently falling back to IP-keyed throttling.

## Decisions Made

- **RuntimeException over abort(500) for V5-059/V5-060:** `\RuntimeException` is unhandled and gets captured by Nightwatch as a genuine exception with a stack trace, which is exactly the "loud failure" the audit wanted. `abort(500)` would produce an HTTP exception that might be caught silently by some error handlers.
- **health-check limiter respects `$throttleEnabled`:** Kept consistent with all other limiters — disabling throttle in dev/test still works for health checks.
- **No custom 429 response on health-check limiter:** Health endpoints don't need a custom message body; the default Laravel 429 is fine for infrastructure monitoring tools.

## Notes

The `bootstrap` limiter already had `supabase.jwt` as the first middleware on its route (line 83 of api.php), so the assert is belt-and-suspenders there. The professional and staff route groups confirmed the same pattern — JWT middleware always precedes throttle middleware.
