# #3-03 â€” Stripe webhook event idempotency uses `insertOrIgnore` without distinguishing race-loss from duplicate

## Plain English

When Stripe sends the same webhook twice in quick succession (which it does during retries), the old code used a database operation that couldn't tell the difference between "I just inserted this for the first time" and "someone else already inserted it a millisecond ago." Both cases looked identical, so there was a theoretical window where two simultaneous requests could both believe they were first. Switching to `firstOrCreate` gives back a clear yes/no: `wasRecentlyCreated` is `true` only for the one request that actually created the row, so only that request processes the event.

## Technical Summary

**Files changed:**
- `app/Http/Controllers/Api/Webhooks/StripeWebhookController.php` â€” replaced `DB::table('billing.webhook_events')->insertOrIgnore([...])` with `WebhookEvent::firstOrCreate(['stripe_event_id' => $event->id], [...])` + `wasRecentlyCreated` guard; payload set via `forceFill` post-create to preserve `$fillable` restriction
- `app/Http/Controllers/Api/Webhooks/StripeConnectWebhookController.php` â€” same change; removed now-unused `use Illuminate\Support\Str`
- `app/Models/Billing/WebhookEvent.php` â€” added `booted()` hook for UUID auto-generation (`static::creating`), added `$timestamps = false` (table uses `processed_at` not Laravel's `created_at`/`updated_at`), updated class docblock

**New contract:** `WebhookEvent::firstOrCreate` is atomic on the `UNIQUE stripe_event_id` constraint. Only the request that sees `wasRecentlyCreated = true` runs the handler; all others return `['received' => true]` immediately.

## Decisions Made

- **UUID in `booted()` not via `forceFill` in controller**: The test schema creates `id TEXT PRIMARY KEY` with no default (unlike production which has `DEFAULT gen_random_uuid()`). Moving UUID generation into the model's `creating` hook makes it work across both environments without touching the test setup.
- **`$timestamps = false` on model**: Eloquent's `firstOrCreate` goes through the full `create()` path, which appends `created_at`/`updated_at`. The table has neither column â€” it uses `processed_at` â€” so the flag is required and was simply missing before (never noticed because the old code used raw `DB::table()` which bypasses Eloquent timestamps).
- **`payload` via `forceFill` after create, not in `firstOrCreate` attributes**: Maintains the existing security invariant (payload not mass-assignable) documented in the model comment. Payload is nullable so a brief window where the row exists without it is safe.

## Notes

Discovered that `WebhookEvent` had no `$timestamps = false` despite the table lacking those columns â€” the raw `DB::table()` approach masked the gap. This was caught immediately by the test suite on the first run.
