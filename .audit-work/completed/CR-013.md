# #CR-013 — Window-keyed caching of window-independent payout/grace state — same data duplicated across every viewed window

## Plain English

The affiliate analytics dashboard has two blocks that show "your next payout" and "you're at risk of losing earnings." These don't change based on which date range you're looking at — they always reflect right now. But they were being stored in cache as if they did depend on the date range, so the same data was duplicated under every date window the affiliate had viewed, and after a payout voided, old windows would still show the outdated urgent banner for up to 5 minutes. Now those two blocks are cached once per affiliate (no date range in the key), so switching windows reuses the same fresh entry and a payout status change propagates everywhere in one TTL cycle.

## Technical Summary

**Files changed:**
- `app/Services/Cache/CacheKeyGenerator.php` — Added `affiliatePayoutState(string $professionalId): string` returning `"analytics:commerce:affiliate:{$professionalId}:payout-state"`. No date segment; one entry per professional.
- `app/Http/Controllers/Api/Professional/Analytics/AffiliateCommerceAnalyticsController.php` — Split `overview()` from one `rememberLocked` call into two. The windowed key (`affiliateCommerceAnalytics($id, $from, $to)`) now covers only `range`, `granularity`, `totals`, `timeseries`, and `brands`. A second `rememberLocked` under `affiliatePayoutState($id)` covers `payout_summary` and `grace_summary`. Results are merged with `array_merge` before the `success()` response. Response shape is identical to before.

**Behavior change:** Payout/grace state now has a single 5-minute cache entry per affiliate rather than one per viewed date window. Stale urgency banners after a payout void can now only persist for one 5-minute window regardless of how many date ranges the affiliate has previously viewed.

## Decisions Made

- **Same 5-minute TTL for payout state key**: The audit item specified no different TTL. The fix is about key structure (eliminating duplication), not freshness. 5 minutes is consistent with the windowed cache.
- **Variable naming (`$windowed` / `$payoutState`)**: Chosen to make the two-cache structure self-documenting without needing a comment to explain the split.
- **No test changes**: Existing tests use table-name-based DB mocks (`DB::shouldReceive('table')->with(...)`) which intercept calls regardless of which cache closure they appear in. All 1509 tests pass unchanged.

## Notes

The `brands` block stays in the windowed cache — it queries `analytics.brand_affiliate_daily` filtered by `from/to` and is genuinely window-dependent. Only `payout_summary` and `grace_summary` were misplaced.
