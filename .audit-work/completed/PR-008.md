# #PR-008 — Stripe transaction blocks don't set isolation level explicitly

## Plain English

When the system processes commission payouts it opens several database "transaction" blocks to safely move money between wallets and create payout records. The question was: what data-consistency guarantee does each block rely on? Without documentation, a future developer might accidentally change the isolation level or add code that assumes stronger guarantees than are actually provided. We added a short comment above each block stating the expected level (PostgreSQL's default, "READ COMMITTED") and noting that the existing row-lock calls already handle the concurrency concern.

## Technical Summary

File changed: `app/Services/Stripe/CommissionPayoutService.php`.

Added an inline comment immediately before each of the four `DB::transaction()` closures:
- `createPayoutBatch` (line ~159): notes that `lockForUpdate()` on both the accruals and reversals reads prevents concurrent batches from claiming the same ledger entries.
- Wallet-debit block inside `processPayoutBatch` (~line 319): notes the atomic wallet-debit + status-update intent and delegates row-level protection to the nested `lockForUpdate()` inside `debitBrandManualBalancePartial`.
- `debitBrandManualBalancePartial` (~line 520): notes the `lockForUpdate()` on the professional row serialises concurrent debits.
- `creditBrandManualBalance` (~line 552): same reasoning for credits.

No logic, method signatures, or contract changes. All 1489 tests pass.

## Decisions Made

- **Document rather than set explicitly**: Laravel's `DB::transaction()` has no `isolationLevel` parameter; setting it would require a `DB::statement('SET TRANSACTION ISOLATION LEVEL ...')` call inside the closure, which is noisier than a comment and adds a round-trip. Since `READ COMMITTED` is already the PostgreSQL default and `lockForUpdate()` provides the needed safety, documentation is the lower-risk, lower-complexity path.
- **Comment placement above the closure, not inside it**: Keeps the comment co-located with the `DB::transaction()` call itself, where a reader deciding whether to change the level would naturally look.

## Notes

The auditor referenced `DB::transaction(..., isolationLevel: ...)` as a potential fix; this named parameter does not exist in Laravel 12's `DB::transaction()`. The documentation path is the correct resolution for P3.
