# #CR-007 — BrandFundingGate checks on-file column nullability, not Stripe live state — detached card still passes the gate

## Plain English

The brand funding gate was checking whether our database had a card saved for the brand, but not whether that card was still active at Stripe. Now, when a brand removes their card directly in Stripe's dashboard, Stripe sends us a webhook event and we immediately clear the card from our database — so the gate will correctly block the brand on their next invite attempt. We also added logging so every gate block is recorded with the brand ID and reason for auditing purposes.

## Technical Summary

Three files changed:

**app/Http/Middleware/BrandFundingGate.php** — Added `use Illuminate\Support\Facades\Log` and a `Log::warning` call with `brand_id` and `reason` context when the gate denies a brand. Updated the block comment to document that this is an "on-file" check (not a live Stripe authorise) and that correctness depends on the `payment_method.detached` webhook handler proactively nullifying the column.

**app/Http/Controllers/Api/Webhooks/StripeWebhookController.php** — Added `'payment_method.detached'` to the match dispatch table and added a new `handlePaymentMethodDetached(object $paymentMethod): void` private method. The handler looks up the brand by `stripe_payment_method_id = $paymentMethod->id`, nulls the column via `$brand->update(['stripe_payment_method_id' => null])`, and logs the result. It handles missing PM ID and no-match cases gracefully without throwing.

**tests/Feature/Stripe/StripeWebhookPaymentMethodDetachedTest.php** — 4 new Pest tests using the existing reflection-based pattern from `StripeWebhookSubscriptionUpdatedTest`: clears PM on match, preserves customer ID, no-op on unknown PM, no-op on missing PM ID. Uses `ALTER TABLE … ADD COLUMN` with try/catch to extend the shared in-memory professionals table with the Stripe columns.

## Decisions Made

- **Option B (webhook handler) over Option A (live Stripe check):** The existing codebase is event-driven for Stripe state; a live check would add latency to every invite attempt and a Stripe-down dependency. The small delivery window is acceptable given low invite frequency and Stripe's sub-second webhook delivery in practice.
- **Look up brand by stripe_payment_method_id (not previous_attributes.customer):** The PM ID is directly available on the detached event object (`data.object.id`). Navigating `previous_attributes.customer` would work but adds coupling to Stripe's event diff format.
- **Only null stripe_payment_method_id, not stripe_customer_id:** The customer record at Stripe persists after card removal so a new card can be attached to the same customer. Clearing customer ID would break the re-attachment flow.
- **Handler in StripeWebhookController (not StripeConnectWebhookController):** `payment_method.detached` is a billing-level event, not a Connect event. The existing billing webhook endpoint and signature secret are the right home.

## Notes

The Stripe webhook dashboard must be updated to subscribe the billing endpoint to `payment_method.detached` events — this code change alone does not register the event type with Stripe. Without that config change the handler will never fire (default events logged as debug-level "Unhandled Stripe billing event" via the match default).
