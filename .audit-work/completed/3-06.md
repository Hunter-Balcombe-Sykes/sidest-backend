# #3-06 — Stripe rate-limit errors retry without explicit backoff

## Plain English

When Stripe says "slow down" (a 429 rate limit), the payout orchestrator job was relying on a generic retry schedule that wasn't long enough for Stripe rate limits to clear. The fix makes the job wait progressively longer on each rate-limit hit — 2 minutes, then 4 minutes, then up to 8 minutes — before trying again. It also uses a smarter retry method that doesn't count against the job's failure budget, so a temporary Stripe hiccup doesn't waste retry attempts meant for real errors.

## Technical Summary

**File changed:** `app/Jobs/Stripe/ProcessCommissionPayoutsJob.php`

Added `use Stripe\Exception\RateLimitException;` import and wrapped the `$payoutService->processEligiblePayouts()` call in a try/catch block. On `RateLimitException`, the job calls `$this->release($delay)` with `$delay = min(120 * 2^(attempts-1), 600)` — producing delays of 120s, 240s, 480s (capped at 600s) — then returns early. This uses `release()` rather than re-throwing so the attempt is requeued without consuming a failure against `$tries = 3`. The generic `backoff()` schedule `[60, 180]` remains unchanged for non-rate-limit exceptions.

The child job `ExecuteCommissionPayoutJob` already had its own `[60, 120, 300, 600]` backoff for rate limits at the actual Stripe API call sites; this fix adds the same protection at the orchestrator level.

## Decisions Made

- **Used `release()` instead of `throw $e`:** re-throwing would apply the generic `backoff()` `[60, 180]` schedule, which is shorter than the rate-limit-specific delays we want. `release()` also avoids consuming a "failure" attempt against the retry budget for a transient throttle.
- **Delay formula `min(120 * 2^(attempt-1), 600)`:** gives 2min → 4min → 8min, which is meaningfully longer than the generic `[60, 180]` and capped at 10min to avoid excessive queue latency.
- **No change to `$tries = 3`:** adequate; if rate limiting persists across all 3 attempts, `failed()` will surface it to Nightwatch as expected.
- **No change to `ExecuteCommissionPayoutJob`:** out of scope for this audit item; the child job already has appropriate backoff.

## Notes

The orchestrator itself rarely makes direct Stripe API calls (it dispatches child jobs), so this catch is primarily defensive. The more common rate-limit path runs through `ExecuteCommissionPayoutJob` → `CommissionPayoutService`, which already handles `RateLimitException` by re-throwing to trigger the child job's `[60, 120, 300, 600]` backoff.
