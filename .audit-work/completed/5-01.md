# #5-01 — Square/Fresha token refresh has a race window — concurrent refreshes clobber each other

## Plain English

When two background jobs both notice a Square or Fresha access token is about to expire, they used to both rush off to fetch a new one at the same time, then overwrite each other's results — leaving the integration in a broken state with mismatched tokens. Now the first job grabs a lock, does the refresh, and saves the new token. Any other job waiting for the same integration sees the lock is held, waits, and once the lock is released it re-reads the token from the database — if it's already fresh, it returns it without making another HTTP call.

## Technical Summary

Changed `refreshAccessToken` in both `SquareTokenService` and `FreshaTokenService` to wrap the entire refresh flow in `Cache::lock("integration_refresh:{$integration->id}", 30)->block(10)`. After acquiring the lock, `$integration->refresh()` reloads the model from DB; if `expires_at` is now beyond the 5-minute buffer, the method returns the existing token early (single-flight exit). On `LockTimeoutException` (10s wait exceeded), the same re-read + early-return logic runs before re-throwing, so callers that waited long enough to see a completed refresh still succeed. Lock release is always attempted in a `finally` block with a swallowed `Throwable` to match the `CacheLockService` pattern. Added imports: `Cache`, `LockTimeoutException`, `Throwable`.

## Decisions Made

- **Cache::lock over DB lockForUpdate**: Cache lock is simpler, doesn't hold a DB transaction open during the HTTP call (which can run up to 20s), and matches the existing `CacheLockService` pattern in the codebase.
- **Re-throw LockTimeoutException (not a domain exception)**: Job retry infrastructure handles this naturally; converting to `SquareApiException`/`FreshaApiException` would mislead callers about the actual failure mode.
- **Double-check with $integration->refresh()**: Eloquent's `refresh()` reloads all attributes from DB in-place, giving each caller the canonical post-lock state without an extra query or re-instantiation.

## Notes

The lock key `integration_refresh:{uuid}` is shared across both Square and Fresha services, but since each integration has a unique UUID the keys are already namespaced per-provider per-professional. No collision risk.
