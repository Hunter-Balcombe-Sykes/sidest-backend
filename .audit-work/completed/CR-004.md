# #CR-004 — ProfessionalServiceController::restore() repeats the sort_order bug `store()` was fixed for

## Plain English

When a service was soft-deleted, another service could "claim" its position in the ordering list. The code for restoring a deleted service had two problems: it reversed the order of operations (trying to delete-then-reorder instead of reorder-then-undelete), and it was looking at the wrong scope when computing where to place the restored service. The first problem would crash the app; the second would silently put the service in the wrong position. Both are now fixed.

## Technical Summary

**File changed:** `app/Http/Controllers/Api/Professional/ProfessionalSiteSelfManagement/ProfessionalServiceController.php` — `restore()` method.

**Old behaviour:** `$service->restore()` was called first, setting `deleted_at = null` on the row. If another live service held the same `(professional_id, sort_order)` pair, the partial unique index `WHERE deleted_at IS NULL` fired immediately, producing a 500. The subsequent max query also scoped by `category_id`, which is not part of the index, returning an incorrect ceiling value for cross-category collisions.

**New behaviour:** The max sort_order is computed first across all live services for the professional (`whereNull('deleted_at')`, no `category_id` filter — mirrors the `af715f7` fix on `store()`). The new sort_order is written to the still-soft-deleted row via `saveQuietly()` (avoiding observer side-effects on a deleted record), then `restore()` sets `deleted_at = null`. At that point the unique constraint is already satisfied.

**Test added:** `tests/Feature/Professional/ServiceRestoreSortOrderTest.php` — creates the production partial unique index on the test table, soft-deletes a category-A service, inserts a category-B service that claims the freed sort_order slot, restores the first service, and asserts HTTP 200 with `deleted_at = null` and `sort_order = 1`.

## Decisions Made

- `saveQuietly()` instead of `save()` for the interim sort_order update: the row is still soft-deleted at that point; firing Square/Fresha sync observers on a deleted-but-not-yet-restored record could cause spurious sync errors. `restore()` still fires the full model lifecycle events.
- Explicit `->whereNull('deleted_at')` on the max query: redundant with the SoftDeletes global scope, but matches the `store()` fix exactly and makes the constraint-awareness explicit to future readers.

## Notes

SQLite (used in the test environment) requires the schema name on the index identifier, not the table name: `CREATE UNIQUE INDEX IF NOT EXISTS site.services_pro_sort_order_uq ON services (...)` — not `ON site.services (...)`. This is the inverse of the PostgreSQL syntax and caused a "near '.': syntax error" on the first attempt.
