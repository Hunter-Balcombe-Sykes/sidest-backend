# #V5-002 — GDPR webhook deduplication on malformed payload causes permanent stuck state

## Plain English

When Shopify sends a GDPR deletion or data-export webhook, it retries until it gets a successful response. Before this fix, if the very first delivery had a malformed or incomplete payload, the system would still acknowledge it with a 202 and store it in the database. Every subsequent retry (with the same body) was recognized as a duplicate and skipped — so the actual deletion or export job never ran, silently failing GDPR compliance. Now, if the payload is broken, the system rejects it with a 422 so Shopify knows to keep retrying until a valid delivery succeeds.

## Technical Summary

**File changed:** `app/Http/Controllers/Api/Webhooks/ShopifyGdprWebhookController.php`

In `handleGdprWebhook()`, JSON parsing and field validation now happen before `hash('sha256', $rawBody)` and `GdprRequest::firstOrCreate()`. Two failure modes are guarded:
1. Non-JSON body (`json_decode` returns null) → 422, no DB row created
2. Missing required topic fields → 422, no DB row created

A new private method `hasRequiredFields(string $topic, array $payload): bool` encodes the rules: `shop/redact` has no payload field requirements (shop_domain from the header is sufficient); `customers/data_request` and `customers/redact` both require a non-empty `payload['customer']['email']` (both corresponding jobs skip/fail without it).

**File changed:** `tests/Feature/Shopify/Gdpr/GdprWebhookControllerTest.php` — three new test cases added covering malformed JSON body, missing email on customers/redact, and missing email on customers/data_request. All 1502 tests pass.

## Decisions Made

- **No `shop_id` validation for `shop/redact`**: The `RedactShopJob` resolves the integration via `shop_domain` (stored from the header), not from `payload['shop_id']`. Adding `shop_id` as required would have broken two existing tests that send valid shop/redact webhooks without it, and would not have prevented a real job failure.
- **422 not 400 for both error cases**: 422 (Unprocessable Entity) is semantically correct for structurally valid JSON with missing fields. For the malformed-JSON case, 422 is also used for consistency, and Shopify's retry behavior is triggered by any non-2xx response regardless of which 4xx code is returned.
- **`hasRequiredFields` as a private method**: Keeps the guard logic named and testable in isolation rather than embedding conditionals inline.

## Notes

The `shop/redact` topic's `hasRequiredFields` branch unconditionally returns `true`. Its main benefit from this fix is the upstream JSON parse check — a malformed JSON body for shop/redact would previously have been stored as RECEIVED with `$payload = []`, which the `RedactShopJob` would then process with an empty payload. That silent failure path is now rejected with 422.
