## Plain English

The "send a success response" helper in the API base controller was easy to misuse because it looks similar to the error helper but has different parameters — developers kept adding a message argument that got silently misinterpreted as the HTTP status code. We added a unit test that locks in the correct behavior and explicitly documents that passing a string where the status code goes is a TypeError. We also updated the inline comment on the method to call out the exact mistake to watch for.

## Technical Summary

**Files changed:**
- `app/Http/Controllers/Api/ApiController.php` — expanded docblock on `success()` to explicitly document the `(data, status)` signature and warn that `success($data, 'message', 200)` is the common footgun (string coerces to 0, third arg is silently dropped).
- `tests/Unit/Controllers/ApiControllerSuccessHelperTest.php` (new) — four Pest unit tests covering: default 200 response, explicit status code, no-argument call, and the TypeError assertion for non-numeric string passed as `$status`.

**New behavior / contracts:**
- PHP 8.2 already throws `TypeError` at runtime when a non-numeric string is passed to `int $status`. The new test makes this contractual and CI-enforced.
- The test file is named `ApiControllerSuccessHelperTest.php` and lives in `tests/Unit/Controllers/`.

## Decisions Made

- Chose the **unit test** path over the CI-grep or signature-change options. CI config is out of scope for unattended sessions; signature change adds complexity. The test is low-blast-radius and becomes a permanent regression guard.
- Did not add `declare(strict_types=1)` to `ApiController.php` — that only tightens type coercion for calls made *within that file*, not callers in subclasses, so it would have given false confidence.
