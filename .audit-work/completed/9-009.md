# #9-009 — InviteExpirySweepJob loads all expired invites into memory

## Plain English
The daily job that marks expired affiliate invites used to pull every expired invite into memory all at once before processing them. If there were thousands of expired invites, this could cause the job to run out of memory and crash. The fix processes them in batches of 500 at a time, so memory usage stays flat no matter how many invites accumulate. The job also now gets more time and retry attempts before giving up.

## Technical Summary
Changed `InviteExpirySweepJob::handle()` in `app/Jobs/Notifications/InviteExpirySweepJob.php` to replace `DB::table(...)->get()` + `foreach` with `->chunkById(500, closure)`. The closure closes over `$publisher` and `$now` from the outer scope. Per-invite try/catch remains unchanged, preserving fault isolation within each chunk. Class properties bumped: `$timeout` 120 → 300, `$tries` 1 → 3.

## Decisions Made
- Chunk size of 500: reasonable middle ground — small enough to cap memory, large enough to minimize per-chunk query overhead on a table expected to stay in the thousands range.
- Kept `$now` captured at handle() entry rather than re-capturing inside the chunk closure, so all rows in a single sweep share the same expiry timestamp boundary (avoids edge cases where rows just crossing the boundary mid-sweep get inconsistent treatment).

## Notes
`chunkById` on the query builder uses keyset pagination (`WHERE id > ?`) rather than OFFSET, so it stays performant even if the expired-invite set is large. The `id` column was already in the original select list, satisfying `chunkById`'s requirement.
