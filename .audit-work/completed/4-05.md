# #4-05 — shop/update webhook overwrites brand-customized fields

## Plain English

When Shopify sends a "shop updated" notification, Side St now checks whether the brand has marked any of their profile fields as "locked" before overwriting them. A locked field (like display name or phone number) is one the brand wants to manage locally — Shopify's value is ignored for that field while everything else still syncs normally. By default, no fields are locked, so existing brands see no change in behavior.

## Technical Summary

- **`ShopProfileAutoFillService.resyncFromShopData`** (`app/Services/Shopify/ShopProfileAutoFillService.php`): Added a locked-fields check at the top of the field-map loop. Fields whose column name appears in `integration.provider_metadata.shopify_sync_locked_fields` are added to `$preserved` and skipped — no Shopify value is written. A private `getLockedFields(ProfessionalIntegration): string[]` helper reads and validates the metadata key (defaults to `[]`).
- **`ProcessShopifyShopUpdateJob.handle`** (`app/Jobs/Shopify/ProcessShopifyShopUpdateJob.php`): Changed from calling `fillFromShopData` (the signup/empty-fill path) to `resyncFromShopData` (the correct resync path, which now honours locked fields). Also added an early-return guard when no Shopify integration record exists, with a warning log. Removed the now-unused `BrandProfile` fetch and import.
- **`ShopifyResyncTest.php`** (`tests/Feature/Professional/Store/ShopifyResyncTest.php`): Three new tests — single locked field is preserved, non-locked fields still sync alongside a locked one, and multiple fields across `professional` and `brand_profile` targets can be locked simultaneously.

No schema migration required — `shopify_sync_locked_fields` lives in the existing `provider_metadata` JSONB column on `professional_integrations`.

## Decisions Made

- **Lock storage location — `provider_metadata` JSONB vs. new DB column**: Used existing `provider_metadata` on `professional_integrations`. No migration needed, is naturally Shopify-scoped, and is consistent with how other Shopify-specific config (`shop_currency`, `storefront_access_token`, etc.) is stored.
- **Granularity — per-field array vs. all-or-nothing boolean**: Chose per-field array (`shopify_sync_locked_fields: string[]`). Matches the audit's phrasing ("fields"), is more flexible without adding complexity, and is backward-compatible (empty array = all fields sync).
- **Default — opt-in vs. opt-out**: Default is empty array (all fields sync). Preserves current behavior for all existing integrations; brands explicitly lock fields they want to manage locally.
- **Fixing the wrong method call in the job**: The job was calling `fillFromShopData` (documented as "signup path, only fills empty fields") but the code actually overwrites. Switching to `resyncFromShopData` makes the call semantically correct and places all resync logic (including locked-field handling) in one method.

## Notes

The `fillFromShopData` → `fillProfessional` code has a subtle inconsistency: its docblock says "only fills empty fields" but the implementation uses `?: existing` which overwrites if Shopify has a value. This wasn't fixed here to keep blast radius minimal — the audit item was specifically about the webhook path. Worth a follow-up tidy.
