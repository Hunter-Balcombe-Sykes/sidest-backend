# #9-008 — BrandAffiliateInviteObserver fetches names per event — N+1 on bulk invite

## Plain English

When a brand uploads a CSV of 100 affiliate invites, the system was hitting the database 100 separate times just to look up the brand's display name — once for every invite created. The fix stores the name the first time it's fetched and reuses that stored value for all subsequent invites in the same batch, so the database is only queried once no matter how many invites are created at once.

## Technical Summary

- **File changed:** `app/Observers/Core/BrandAffiliateInviteObserver.php`
- Added `private static array $brandNameCache = []` class property.
- `brandName(string $brandProfessionalId)` now checks `self::$brandNameCache[$brandProfessionalId]` before issuing a DB query; on a cache miss it queries `core.professionals` and stores the result.
- The static scope ensures the cache is shared across all observer invocations within the same PHP process/request, covering bulk CSV import flows where `created()` fires N times for the same brand.
- No changes to `claimedName()` — that method is called from `updated()` events (accept/decline), which are single one-at-a-time user actions and not a batch concern.

## Decisions Made

- **Static vs. instance cache:** Used `static` property rather than `$this->` because Laravel's IoC container could resolve a new observer instance per event in some configurations. Static guarantees cross-instance sharing within the same process.
- **Scope of fix to `brandName()` only:** The audit item identified bulk `created` events as the source; `claimedName()` on `updated` is not a batch scenario so was left untouched to minimize blast radius.

## Notes

No surprising findings. The fix is purely additive (memoization layer) with no behavioral change — the DB query logic, fallback values, and `deleted_at` filter are unchanged.
