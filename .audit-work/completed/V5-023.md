# #V5-023 — WebhookEvent payload mass-assignable, no schema validation

## Plain English
The WebhookEvent model (our Stripe event log) had its `payload` field listed as mass-assignable, meaning any code that used Laravel's convenient bulk-fill feature on this model could store arbitrary data in that column. We removed it from that list. The column still gets written — but only through a lower-level database call that already sits behind Stripe's HMAC signature verification, so nothing legitimate was broken.

## Technical Summary
- **File changed:** `app/Models/Billing/WebhookEvent.php`
- Removed `'payload'` from `$fillable`; `stripe_event_id`, `event_type`, and `processed_at` remain.
- Both insertion sites (`StripeWebhookController` and `StripeConnectWebhookController`) use `DB::table('billing.webhook_events')->insertOrIgnore()`, which bypasses `$fillable` entirely — so runtime behavior is unchanged.
- Upstream guards (Stripe HMAC via `Webhook::constructEvent()` + `ValidatesStripeWebhookPayload::validateEventStructure()`) already enforce that only verified Stripe payloads reach the insert.
- Added a class-level comment documenting the intentional insertion path and why `payload` is excluded from mass-assignment.

## Decisions Made
- **No controller-level schema validation added**: The existing HMAC verification + structural check (`id`, `type`, `data`, `data.object` present) already constrain the payload to verified Stripe event objects. Adding JSON schema validation on top would be redundant defense-in-depth with no practical benefit.
- **No new test added**: The fix closes a theoretical Eloquent path with no current callers; all 1506 existing tests pass unchanged and the webhook integration tests already cover the live insertion path.

## Notes
The `DB::table()` bypass is intentional — it allows `insertOrIgnore` to work atomically without triggering model events or casts that could interfere with the idempotency guarantee. The `payload` cast to `array` on the model is still useful for read queries (e.g., debugging via tinker), even though it doesn't affect the write path.
