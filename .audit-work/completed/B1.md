# B1 — Missing failed() handlers across financial / onboarding jobs

## Plain English

Five jobs that handle money and onboarding were missing an alarm for when they give up after repeated failures. Now each one calls `report()` (which notifies Nightwatch) and writes a structured log entry with enough context to identify exactly what failed and for whom. The weekly analytics notification job also gets a second attempt before giving up, which is safe because it already deduplicates by design.

## Technical Summary

Added `public function failed(\Throwable $e): void` to five job classes:

- `app/Jobs/Stripe/VoidPendingCommissionsForLinkJob.php` — logs `affiliate_professional_id`, `brand_professional_id`, `reason`
- `app/Jobs/Shopify/ProcessShopifyOrderWebhookJob.php` — logs `brand_professional_id`, `shopify_event_id`, `shopify_order_id` (from `Arr::get($this->orderPayload, 'id', '')`)
- `app/Jobs/Shopify/ProcessShopifyOrderUpdatedWebhookJob.php` — logs `professional_id`, `topic`, `shopify_order_id` (resolved via `Arr::get($this->payload, 'id') ?? Arr::get($this->payload, 'order_id', '')` to handle both standard topics and `refunds/create` which uses `order_id`)
- `app/Jobs/Store/SeedAffiliateDefaultSelectionsJob.php` — logs `affiliate_professional_id`, `brand_professional_id`
- `app/Jobs/Notifications/SendWeeklyAnalyticsNotificationJob.php` — logs `error` message; also raised `$tries` from 1 to 2

All handlers follow the `report($e); Log::error(...)` pattern established by `ExecuteCommissionPayoutJob`, `ProcessCommissionPayoutsJob`, and `VoidExpiredPayoutsJob`.

## Decisions Made

- **`refunds/create` order ID resolution in `ProcessShopifyOrderUpdatedWebhookJob::failed()`**: used `Arr::get($this->payload, 'id') ?? Arr::get($this->payload, 'order_id', '')` rather than a topic-specific branch — simpler and correct because for `refunds/create` the `id` key is the refund ID (which is still useful for debugging), while `order_id` is the parent order. The log field is labeled `shopify_order_id` but in the refund case it will contain the order's ID via the fallback.
- **`$tries = 2` for `SendWeeklyAnalyticsNotificationJob`**: confirmed idempotency via the `"analytics.weekly.{$professional->id}.{$yearWeek}"` dedupe key passed to `NotificationPublisher::publish()`, which uses `insertOrIgnore` semantics. A second run of the same week is safe.
- **No `$tries` increase for other jobs**: the other four jobs already had `$tries = 3`; no change needed.

## Notes

`SeedAffiliateDefaultSelectionsJob` already had an internal `try/catch` in `handle()` that logged and re-threw — the new `failed()` provides the terminal handler for after all retries are exhausted, which was the gap.
