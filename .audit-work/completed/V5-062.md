## Plain English

Previously, different API endpoints returned different sets of fields for a Professional — some returned raw database model objects with all fields exposed, while others returned hand-crafted arrays that each made their own choices about what to include. This made the API unpredictable and risked leaking internal fields (like the auth user ID or payment status) to the wrong audience. We've now defined three canonical resource shapes — one for each audience — so every endpoint consistently returns exactly the fields that audience should see.

## Technical Summary

**New files:**
- `app/Http/Resources/ProfessionalDashboardResource.php` — own-profile shape for the authenticated professional; includes `auth_user_id`, full profile fields, `stripe_connect_status`, and Square integration fields (`square_connected`, `square_merchant_id`) conditionally via `whenLoaded('squareIntegration')`.
- `app/Http/Resources/ProfessionalStaffResource.php` — staff admin shape; includes `auth_user_id` and full profile fields, but excludes payment integration fields irrelevant to staff management workflows.
- `app/Http/Resources/ProfessionalPublicResource.php` — public-safe subset: `id`, `handle`, `handle_lc`, `display_name`, `bio`, `professional_type`, `public_contact_number`, `public_contact_email`, `location_city/state/country`. Excludes all PII (`primary_email`, `phone`, `street_address`, `auth_user_id`, internal status fields).

**Updated controllers:**
- `ProfessionalController::show()` — loads `squareIntegration` relation, replaces 30-line inline array with `ProfessionalDashboardResource`.
- `ProfessionalController::update()` — replaced `ProfessionalResource` with `ProfessionalDashboardResource`.
- `StaffProfessionalController::show()` — replaces 20-line inline array with `ProfessionalStaffResource`.
- `StaffProfessionalController::updateStatus()`, `update()`, `restore()` — replaced raw `$professional->fresh()` with `ProfessionalStaffResource`.
- `BootstrapController::bootstrap()` — replaced raw `$professional->fresh()` in transaction return with `ProfessionalDashboardResource`.

**Unchanged:** `StaffProfessionalController::index()` deliberately retains its lightweight inline map (intentionally slim for list-view performance). `ProfessionalResource` retained as-is (no callers remain after migration).

## Decisions Made

- **Square fields via `whenLoaded`** rather than always fetching: the `show()` endpoint explicitly loads `squareIntegration` so the fields appear there; `update()` and `bootstrap()` don't load it, so those responses omit Square fields — consistent with prior behavior.
- **`ProfessionalPublicResource` created but not wired to a controller**: the public site payload comes from a database view (`public_site_payload`), not a direct Professional model return, so there's no controller to immediately migrate. The resource class establishes the canonical shape for future use.
- **Staff index left as inline map**: it's a paginated list view where the comment explicitly says "Keep response light for list-view" — introducing a resource class here would be over-engineering and could silently cause N+1 issues on eager-loaded relations.
