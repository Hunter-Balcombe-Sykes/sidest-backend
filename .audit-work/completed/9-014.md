# #9-014 — Horizon analytics worker memory limit may be tight for aggregations

## Plain English

The worker that handles analytics and image jobs had a memory cap of 256 MB. When a worker process exceeds its cap, Horizon restarts it gracefully — but if the cap is too low during large analytics rebuilds or image transformations, workers restart in loops and jobs take longer to complete. We raised the limit to 512 MB to give those workers enough headroom as data volumes grow.

## Technical Summary

Changed `config/horizon.php` `supervisor-analytics.memory` from `256` to `512`. This is Horizon's soft memory threshold: when a worker process exceeds this value (in MB) after completing a job, it exits gracefully and Horizon spawns a replacement. The analytics jobs (`RebuildCommerceDailyAggregatesJob`, hourly variants, booking/site equivalents) use DB-side aggregation and are not heavy memory consumers by themselves — the real pressure comes from the `images` queue sharing this supervisor, where PHP image libraries can spike heap during transformation. Added an inline comment documenting the rationale.

## Decisions Made

- Set to 512 (not higher): the audit item specified 512 as the target; analytics jobs use DB-level aggregation so they won't spike PHP memory on their own, and 512 is a reasonable ceiling for image processing jobs on this hardware class.
- Added explanatory comment: the original comment didn't mention memory rationale; future readers need to know why this supervisor's limit differs from the others (128 MB each).

## Notes

Exploration confirmed all six analytics jobs delegate aggregation to `CommerceAnalyticsAggregateService` / equivalent services using `DB::raw()` with `SUM`/`COUNT`/`GROUP BY` — application memory per job is bounded by the number of currency codes (typically 1–4 rows). The `images` queue on the same supervisor is the more plausible memory spike source.
