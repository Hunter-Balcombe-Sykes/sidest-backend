# #PR-007 — Shopify webhook `fallback_secret` has no rotation deadline

## Plain English
When Shopify webhook secrets are rotated, the app keeps the old secret as a "fallback" so webhooks signed just before the rotation still pass. The problem was nothing told developers to eventually remove that fallback secret — it could sit there forever, reducing the security benefit of rotating. We added a comment directly in the code making the rule explicit: clear the fallback within 30 days of rotating the primary secret.

## Technical Summary
Changed: `app/Http/Controllers/Concerns/ValidatesShopifyWebhookHmac.php`

Added a two-line inline comment above the `services.shopify.fallback_secret` config read inside `isValidShopifyHmac()`. The comment documents the intent of the dual-secret pattern (zero-downtime rotation window) and sets an explicit 30-day maximum retention deadline for the fallback, after which it must be cleared from env/config.

No logic changes; no new tests required — existing Shopify HMAC tests continue to cover the validation path.

## Decisions Made
- **30-day rotation window:** Standard practice for webhook secret rotation; long enough for all in-flight webhooks to drain, short enough to bound the exposure window if the old secret is ever compromised.
- **Comment placement:** Inline above the fallback_secret line rather than on the trait docblock, so the constraint is collocated with the code it governs and visible during code review.

## Notes
No surprising findings. The fix is purely documentary — no logic, config, or test changes were needed.
