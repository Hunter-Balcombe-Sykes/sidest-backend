## Plain English

When a professional manually deleted a service in Side St, the next Square full sync would find it (because the sync looks up soft-deleted records by Square ID), fill in Square's data, and bring it back to life as if nothing happened. The fix adds a "who deleted this?" marker to every service. Sync-driven deletions are now tagged `square`; anything without that tag (including every existing deletion) is treated as a manual deletion and left alone during sync.

## Technical Summary

**Files changed:**
- `supabase/migrations/20260504100000_add_deleted_origin_to_services.sql` — adds `deleted_origin varchar(16) NULL` to `site.services`. Null = manually deleted; `'square'` = deleted by Square catalog sync.
- `app/Models/Core/Professional/Service.php` — adds `deleted_origin` to `$fillable`.
- `app/Services/Square/SquareServiceSyncService.php` — three changes:
  1. **Upsert loop** (the zombie path): before filling a trashed service, checks `deleted_origin !== 'square'` and `continue`s, leaving the service deleted and unmodified.
  2. **Incremental deletion** (Square reports `deleted: true`): sets `deleted_origin = 'square'` before saving/deleting.
  3. **Full-sync cleanup** (services absent from Square snapshot): sets `deleted_origin = 'square'` before saving/deleting.
- `tests/Pest.php` — adds `setupServicesTable()` and `setupServiceCategoriesTable()` SQLite helpers.
- `tests/Feature/Services/Square/SquareServiceSyncServiceTest.php` — 4 new tests covering: manually-deleted stays deleted, Square-deleted restores on re-sync, incremental delete stamps origin, full-sync cleanup stamps origin.

**New behavior:** All Square-driven deletions are now stamped `deleted_origin = 'square'`. Services deleted via the Side St UI (or via API) have null origin and are permanently shielded from Square sync restoration within the soft-delete retention window.

## Decisions Made

- **Null = manual** (safe default): existing soft-deleted rows (origin unknown) inherit null and will not be auto-restored. This is the conservative choice — a service that was previously deleted by sync might need one more manual re-sync trigger to come back, which is better than silently restoring something a professional deliberately deleted.
- **No `deleted_by` audit user tracking**: the column is intentionally minimal — it tracks system origin (`'square'`), not actor identity. Actor tracking is a separate concern covered by the deletion audit table.
