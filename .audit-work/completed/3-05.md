# #3-05 — Wallet debit and PaymentIntent create are not in a single transaction

## Plain English

When a payout job crashes repeatedly (say, Stripe keeps timing out) and Horizon gives up, the job's cleanup handler now automatically returns the brand's wallet balance to what it was before the payout attempt. Previously, the wallet stayed debited even though no charge was ever completed, and someone on the team would have had to manually fix it. Now the cleanup is automatic, and if staff want to retry the payout, it starts fresh with the correct balance.

## Technical Summary

**Files changed:**
- `app/Services/Stripe/CommissionPayoutService.php` — changed `creditBrandManualBalance()` visibility from `private` to `public` so the job's `failed()` hook can resolve and call it via `app()`.
- `app/Jobs/Stripe/ExecuteCommissionPayoutJob.php` — overhauled `failed()`: when the payout status is `'collecting'` and `wallet_debit_cents > 0`, it resolves `CommissionPayoutService` from the container and calls `creditBrandManualBalance()` to restore the brand's balance, then clears `wallet_debit_cents = 0` in the same DB write that marks `status = 'failed'`. Clearing `wallet_debit_cents` is load-bearing: `retryPayout()` uses that field to decide whether to resume from `'collecting'` (skip debit) or `'pending'` (fresh debit). Payouts stuck in `'transferring'` (PI already succeeded) are intentionally left unchanged — the wallet debit is still needed there for reconciliation.
- `tests/Feature/Stripe/ExecuteCommissionPayoutJobTest.php` — added 4 new test cases covering: `creditBrandManualBalance` is called with correct args, `wallet_debit_cents` is cleared, wallet is NOT credited for `'transferring'` status, wallet is NOT credited when `wallet_debit_cents = 0`. Updated the existing `failed()` test to also assert `wallet_debit_cents = 0`. Updated the `retryPayout()` test comment to clarify the scenario (transferring-stage failure, not collecting-stage).

## Decisions Made

- **`'transferring'` status left unchanged in `failed()`**: The PI succeeded in that case so the brand was actually charged; crediting the wallet without refunding the PI would leave the balance inconsistent. This matches the existing `transfer_failed_refund_needed` flag pattern for manual reconciliation.
- **Clearing `wallet_debit_cents` on reversal**: Required for correct `retryPayout()` behavior — if the wallet was restored but `wallet_debit_cents` still shows > 0, retryPayout resumes from `'collecting'` and skips the debit against the now-restored balance, causing a double-dip.
- **`creditBrandManualBalance` made `public` rather than extracting a new method**: The existing method is self-contained with correct locking. Making it public is the minimal change with no behavioral risk.

## Notes

The existing code already correctly handles the common crash scenario via Horizon retries: if the process dies mid-PI-creation, the idempotency key causes Stripe to return the same PI on the next attempt. The gap was specifically the final-failure path (all retries exhausted while status = 'collecting') which bypasses the inline error handling in `processPayoutBatch()`.
