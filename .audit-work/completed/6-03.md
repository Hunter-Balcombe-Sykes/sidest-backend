# #6-03 — Click-recording endpoint has no bot/fraud detection beyond IP throttle

## Plain English

Before this fix, anyone who knew the click-tracking URL could hit it as fast as the IP rate limit allowed and every request would count as a real click — inflating link analytics for competitors or testing bots. Now the endpoint checks whether the request looks like it came from an automated tool: if the User-Agent matches a known bot (Googlebot, curl, headless browsers, etc.), the request gets a silent fake-success response and nothing is written to the database. It also validates that any referrer URL is a real URL before storing it, so garbage strings don't pollute the analytics dashboards. A new per-link rate cap (5 clicks per minute per IP per link) adds a second layer of protection even for bots clever enough to spoof their User-Agent.

## Technical Summary

**Files changed:**

- `app/Http/Controllers/Concerns/DetectsClientInfo.php` — Added `isBotUserAgent(?string $ua): bool` method with a 20-signal denylist covering: generic bot/spider/crawler, SEO crawlers (AhrefsBot, SemrushBot, MJ12Bot, DotBot, RogerBot), social crawlers (facebookexternalhit, TwitterBot, LinkedInBot, PinterestBot), search engines (YandexBot, Baiduspider, Slurp), CLI tools (python-requests, python-urllib, curl/, wget/, libwww-perl), and headless/automation browsers (HeadlessChrome, PhantomJS, Puppeteer, Playwright, Selenium). Null/empty UA returns `true` (treated as bot).

- `app/Http/Controllers/Api/PublicSite/AnalyticsController.php` — In `click()`: (1) After `is_published` check but before any block DB query, calls `$this->isBotUserAgent()` and returns a fake 200 response (no DB write) if matched. (2) Sanitizes `$rawReferrer` via `filter_var(FILTER_VALIDATE_URL)` before passing to the `LinkClick` closure; invalid URLs become `null`.

- `app/Providers/AppServiceProvider.php` — Registered `analytics-click` rate limiter: `Limit::perMinute(5)->by($request->ip() . ':click:' . $blockId)`. Respects the global `SIDEST_THROTTLE_ENABLED` flag.

- `routes/api/publicSite.php` — Click route now carries `['throttle:analytics', 'throttle:analytics-click']` (both the global 120/min IP limiter and the new 5/min per-link limiter).

- `tests/Feature/Analytics/ClickBotFilterTest.php` — Six new Pest tests: Googlebot UA → 200 no DB record, HeadlessChrome UA → 200 no DB record, curl UA → 200 no DB record, legitimate mobile Safari UA → 201 with DB record, malformed referrer → 201 with `referrer=null` in DB, valid referrer → 201 with referrer preserved.

## Decisions Made

- **Silent 200 for bots (not 204 or 422):** Returning a different status code would let an attacker fingerprint the filter and iterate on UAs until they find one that returns 201. A fake 200 with `{message: "Click recorded"}` is indistinguishable from a successful non-recorded call.

- **Null UA treated as bot:** No legitimate browser omits the User-Agent header. Null UA is the most suspicious signal possible and safe to block.

- **Referrer sanitize-not-reject:** A malformed referrer doesn't mean the click is fraudulent (e.g., a mobile app deep link might produce a weird referrer value). The click is recorded but the bad referrer is dropped — dashboard data stays clean without false-negative filtering.

- **Per-link cap at 5/min:** Generous enough for burst legitimate traffic (e.g., a link shared in a chat group), restrictive enough to limit sustained single-link spam from a single IP. Can be tuned via the rate limiter registration.

- **Dedup window kept at 3 seconds:** The audit said "consider session_id dedup" — this was already implemented at 3 seconds. Extending the window would have broken the existing `ClickDedupTest` (which seeds a "stale" click at 4 seconds and expects a new record). Left as-is and noted in this record.

- **Referrer-domain allowlist not implemented:** The audit called for a check against "brand storefront and registered social handles," but neither field exists on the Site, Professional, or BrandProfile models. The data model doesn't support this without a schema change. Sanitization (valid URL or null) is the defensible pilot-stage implementation.

## Notes

The `DetectsClientInfo::detectDeviceType()` method already returns `'bot'` for UAs containing `bot/spider/crawler`, but it only classifies for storage — it never blocks. The new `isBotUserAgent()` is a parallel method that gates recording entirely. The two methods intentionally have separate concerns: `detectDeviceType` classifies what passed through; `isBotUserAgent` gates the door.

The Symfony test client sends `User-Agent: Symfony` by default, which doesn't match any signal in the denylist — this is why the six existing `ClickDedupTest` cases continue to pass without modification.
