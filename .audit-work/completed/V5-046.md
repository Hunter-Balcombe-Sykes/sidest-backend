## Plain English

Previously, when someone uploaded a video that was too long, the platform would accept the upload, save the file to storage, and only discover the problem once a background worker started processing it — wasting compute time and leaving the user with a stuck "processing" item. Now the duration limit is checked the moment the file arrives, before anything is saved, so the user gets an immediate error message and no resources are wasted.

## Technical Summary

**Files changed:**
- `app/Services/Media/VideoVariantService.php` — extended `probeAndValidate()` to extract duration from the ffprobe result and throw a `RuntimeException` if it exceeds `sidest.video_max_duration_seconds` (default 300s). The existing duration check in `processVariants()` is retained as a defensive backstop.
- `tests/Feature/MediaUploadFailureHandlingTest.php` — added test case `'returns 422 and creates no DB row when video exceeds maximum duration'` that mocks `probeAndValidate()` throwing a duration error and asserts a 422 response with no persisted DB row.

**New behavior:** `probeAndValidate()` now enforces both stream validity and duration limit. The controller's existing try/catch at upload time already converts `RuntimeException` from this method to a 422 `'Invalid video file: ...'` response.

## Decisions Made

Kept the duration guard in `processVariants()` as a backstop rather than removing it. The cost of the extra check there is negligible (it uses the probe result already computed at the top of that method), and it protects against any future code path that might invoke `processVariants()` without going through the upload controller.
