# #CR-009 — At-risk amount selected by exact-timestamp equality; ties produce non-deterministic financial figure

## Plain English

When affiliates haven't connected Stripe yet, the dashboard shows a grace banner with the dollar amount of their earliest-expiring unpaid commission. The bug: if two commissions were created together (in a batch), they'd share the exact same expiry timestamp. The code picked one of them at random, so the dollar figure shown could change between page loads. The fix makes the selection deterministic — when there's a tie, it now consistently shows the highest-value payout amount.

## Technical Summary

**File changed:** `app/Http/Controllers/Api/Professional/Analytics/AffiliateCommerceAnalyticsController.php` — `buildGraceSummary()` method (lines 275–280).

The second DB round-trip that fetches `net_payout_cents` for the earliest-expiring payout now includes `->orderBy('net_payout_cents', 'desc')` before `->first()`. This maps to `ORDER BY net_payout_cents DESC LIMIT 1` in PostgreSQL, making tie-breaking deterministic at the database level. No schema changes, no contract changes to the response shape — `earliest_at_risk_amount_cents` now returns a stable value when multiple payouts share the same `void_at` timestamp.

## Decisions Made

- **Tie-break direction (desc vs asc):** Chose `desc` (highest value wins) over `asc` (lowest value). Reasoning: showing the largest at-risk amount creates appropriate urgency for the affiliate to connect Stripe. The audit item explicitly suggested `desc`.
- **Simple ordering vs window function:** Kept the two-round-trip approach with added ordering rather than rewriting to a window function. The window function approach (mentioned in the audit as also addressing #CR2-007) would be a larger refactor. This fix is minimal-blast-radius and sufficient to eliminate non-determinism.

## Notes

The audit item notes this also relates to #CR2-007 (round-trip count). That optimization remains available as a future improvement — collapsing both queries into a single window-function query would fetch the aggregate and the earliest payout's amount in one shot. Not done here to keep blast radius minimal.
