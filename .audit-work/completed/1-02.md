# #1-02 — Inline `abort(403, ...)` patterns bypass policy system and CI guard

## Plain English

Eight controllers had bulk-id reorder/update endpoints that responded with HTTP 403 ("Forbidden") when the frontend submitted an ID the user didn't own. That status code was misleading — these calls aren't denying permission, they're rejecting stale or invalid input from the client. Five other controllers in the same codebase already used 422 ("Unprocessable Entity") for the identical pattern, so the eight outliers were drift, not intentional design. Converted all eight to `abort(422, 'One or more <X> are invalid.')`, matching the existing precedent. Frontends can now distinguish "the ID you sent is no longer valid, refresh your state" (422) from "you don't have permission to perform this action" (403).

## Technical Summary

The audit item originally prescribed a two-step fix: (a) broaden CI to catch `abort(403, ...)`, and (b) refactor each call into a Policy via `$this->authorizeForUser()`, gated on prerequisite item `#1-01` (the policy rollout) landing first.

The session paused on a question (`questions/1-02.md`) raising three blockers:
- **Q1 (CI broadening):** Rule 4 forbids modifying `.github/workflows/ci.yml` from inside an unattended runner.
- **Q2 (#1-01 dependency):** `#1-01` had not been started; absorbing it would balloon scope.
- **Q3 (semantics):** All eight calls fire only when the user submits an ID their own owned-set query did NOT return — a stale-frontend-state validation error, not authorization.

The answer: **skip CI, do not absorb #1-01, and reclassify the eight calls as 422 validation errors** — which decouples this item from `#1-01` entirely.

## Decisions Made

- **422 not 403, no policies introduced.** Per the question's answer and confirmed by precedent: `ProfessionalServiceController`, `ProfessionalServiceCategoryController`, `StaffServiceCategoryManagementController`, `StaffServiceManagementController`, and `AffiliateProductController` already used `abort(422, 'One or more <X> are invalid.')` for the same bulk-id-validation pattern. The eight converted today now match that house style.
- **Did not touch CI in the unattended run** (`.github/workflows/ci.yml`). Rule 4 of the unattended runner forbids it. CI guard widening was applied as a manual follow-up — see the Notes section below.
- **Did not create policies** (`SiteMediaPolicy`, `BlockPolicy`, etc.). The 422 reclassification eliminates the need for them on these specific call sites. `#1-01` (broader policy rollout) remains independently scoped.
- **Did not modify tests.** Existing reorder tests assert `HttpException` (the parent class), not a specific status code. `abort(422)` and `abort(403)` both throw the same exception, so 1209 tests still pass without test edits.
- **Message wording per controller.** Kept the domain noun ("brand gallery items", "gallery images", "sections", "blocks", "media items", "product photos") so logs and frontend error UX remain readable; harmonized only the suffix to `are invalid`.

## Notes

- **CI gap closed (manual follow-up).** The grep pattern in `.github/workflows/ci.yml:49` was widened from `abort_unless|abort_if` to `abort\(403|abort_unless|abort_if` (post-filtered through `grep "403"` so non-403 `abort_unless/abort_if` calls aren't false-positives). The guard now catches all three forms. Verified locally: 0 matches across `app/Http/Controllers/`. Done outside the unattended run because Rule 4 forbids workflow edits there.
- **`#1-01` dependency removed for these eight call sites.** If `#1-01` (broad policy rollout) is later prioritized, these eight files will already be in the canonical 422 form and need no further action.
- **Tests:** `composer test` → `1209 passed (4001 assertions)` in 25.96s.
