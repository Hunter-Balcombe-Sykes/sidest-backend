## Plain English

When a user uploads a photo or video, their browser tells us the file extension (e.g. `.jpg`, `.mp4`). We were trusting that value and putting it directly into the cloud storage path. Now we check it against a known-safe list (jpg, jpeg, png, webp, mp4, mov, webm) and ignore anything outside that list, falling back to a sensible default instead. This keeps our storage paths clean and predictable regardless of what the browser sends.

## Technical Summary

**Files changed:**
- `app/Services/Media/ImageVariantService.php` — added `ALLOWED_EXTENSIONS` constant and public `safeExtension(string $clientExt, string $default): string` helper that lowercases, strips leading dots, and checks against the allowlist.
- `app/Http/Controllers/Api/Professional/Uploads/ProfessionalUploadController.php:140` — video upload path now calls `$this->mediaService->safeExtension(...)` instead of raw `getClientOriginalExtension()`.
- `app/Services/Media/ImageVariantService.php:218` — `storeOriginal()` (image path) now calls `$this->safeExtension(...)`.
- `tests/Feature/MediaUploadFailureHandlingTest.php` — added `safeExtension` stub to the Mockery mock so the 503 failure-path test continues to pass.

**New behaviour:** Any client-supplied extension not in `['jpg','jpeg','png','webp','mp4','mov','webm']` is silently replaced with the caller-specified default (`'jpg'` for images, `'mp4'` for videos).

## Decisions Made

- Helper placed on `ImageVariantService` (not a static utility) so the allowlist constant and its enforcement are co-located. The method is `public` so the controller can call it without duplicating the allowlist.
- `ltrim($ext, '.')` guards against values like `.jpg` that `getClientOriginalExtension()` should never produce but could in theory if the input came from another path.
