## WHAT TO AUDIT

### 1. Security
- Hardcoded credentials, API keys, tokens, secrets anywhere in code
- Unguarded mass assignment — missing $fillable or $guarded on any Model
- Raw or unsanitised user input passed to queries, shell calls, or external services
- {!! !!} unescaped Blade output — every instance
- Sensitive fields exposed in API responses (passwords, tokens, internal IDs) — check toArray()/toJson()/$hidden on all Models
- Missing CSRF protection on non-API POST/PUT/DELETE routes
- eval(), system(), exec(), shell_exec(), passthru() calls — flag as critical
- serialize()/unserialize() on any user-supplied input — flag as critical
- XML parsing without XXE protection
- Open redirects — any redirect using unvalidated user input as destination
- Insecure direct object references — endpoints where an ID isn't ownership-checked
- JWT or token handling done manually instead of via Sanctum/Passport
- Webhook endpoints with no signature verification (Stripe, GitHub, etc.)
- CORS config allowing wildcard origins on authenticated routes
- Session config missing secure/httponly/samesite flags (config/session.php)
- Missing HTTPS enforcement or HSTS header
- File upload destinations that are publicly accessible when they shouldn't be
- APP_DEBUG or verbose errors potentially exposed in non-local environments
- Auth logic that bypasses standard Laravel Auth — flag exact file and line
- Debug code left in production: dd(), dump(), var_dump(), ray()
- Timing attacks on token or password comparison — any manual string comparison instead of hash_equals()
- Missing account lockout or brute force protection beyond basic rate limiting
- Password reset tokens with no expiry or reuse prevention
- Remember me tokens stored insecurely or with no rotation
- Any endpoint that reveals whether an email/username exists (user enumeration)
- Admin or internal routes not protected by both auth and role/permission middleware
- Sensitive actions (delete account, change email, change password) with no re-authentication requirement

### 2. Input Sanitisation
- Any $request->input() or $request->all() passed to a model, query, or external call without a FormRequest or explicit validate()
- File uploads with no mime type, extension whitelist, or size validation
- String fields with no max length validation writing to DB
- Email, numeric, enum, and date fields with no type validation
- Missing trim/sanitisation on user-facing string inputs stored to DB
- Any form or endpoint accepting input with no validation whatsoever
- Regex patterns used for validation that are vulnerable to ReDoS (catastrophic backtracking)
- HTML content fields stored and rendered without sanitisation (stored XSS risk)
- JSON body parsing with no schema validation on complex nested payloads
- Any input that flows into a file path or system call — flag as critical

### 3. Rate Limiting
- Auth routes with no rate limiter: login, register, password reset, email verify, 2FA
- Public POST endpoints with no throttle middleware: contact forms, webhooks, signups
- File upload endpoints with no rate limit
- Endpoints returning large collections or exports with no rate limit or pagination enforcement
- Admin routes with no stricter rate limit
- Missing or incomplete RateLimiter definitions in RouteServiceProvider
- Rate limiters keyed only by IP — trivially bypassed, flag if no user-based keying on auth routes
- No global fallback rate limit on the API route group

### 4. Supabase & RLS
- Queries that fetch data without scoping to the authenticated user
- Endpoints where any authenticated user could access another user's records
- Any Supabase client call that bypasses RLS (service role key used where anon key should be)
- Features polling Supabase on a schedule that should use Realtime instead
- Any logic that should live in a Supabase Edge Function but runs synchronously in Laravel
- Supabase service role key accessible or logged anywhere outside of config/
- Missing RLS policies inferred from tables referenced in code without user scoping
- Storage bucket access policies — public buckets serving private user files

### 5. Scalability
- Any endpoint that returns an unbounded collection with no pagination
- Queries inside loops — flag every instance, note how many iterations are possible
- N+1 queries — missing eager loading on any relationship traversal
- Missing indexes on: foreign keys, columns used in WHERE clauses, columns used in ORDER BY on large tables
- Synchronous operations blocking the request lifecycle that should be queued: emails, file processing, external API calls, report generation, image resizing
- Jobs with no failed() handler, no retry limit, or no timeout set
- Scheduled commands running heavy queries synchronously — note frequency and estimated cost
- Any feature polling the DB or Supabase on an interval that could be event-driven
- Missing DB transactions around multi-step writes
- Endpoints with no response caching that are called frequently and return largely static data
- Large file uploads streamed entirely into memory rather than to disk/storage
- Any process loading entire Eloquent collections into memory instead of chunking or cursor()
- Queue configuration running synchronously (QUEUE_CONNECTION=sync)
- Anything that would become a bottleneck at 10x current load — note specific file and reason
- Single points of failure — any process with no fallback if an external service is down
- Lack of idempotency on any endpoint that creates or mutates data and could be retried

### 6. Performance & Efficiency
- Repeated identical queries within the same request lifecycle — should be cached or memoized
- Missing query result caching on hot read paths (Cache::remember())
- Eloquent models selecting all columns where only 1-2 are needed — flag for ->select()
- Missing database indexes inferred from query patterns
- Views or API responses computing expensive aggregates on every request with no caching
- Config, translation, or route files not cached — flag if php artisan optimize is not part of deployment
- Autoloaded service providers doing heavy work in register() or boot()
- Middleware running expensive operations on every request unnecessarily
- External HTTP calls with no timeout set
- External HTTP calls with no retry logic or circuit breaker
- Large JSON payloads encoded/decoded repeatedly in a pipeline
- Log calls inside tight loops or high-frequency code paths
- Missing lazy loading or deferred loading on heavyweight service bindings
- Blade templates with logic-heavy @foreach loops that could be precomputed
- Report, export, or data dump endpoints with no chunking, streaming, or background job offload
- Any use of array_map/array_filter on very large datasets that should use lazy collections
- String concatenation inside loops building large outputs — should use implode or streaming

### 7. Error Handling & Observability
- Bare catch blocks that swallow exceptions silently: catch(Exception $e) {}
- Caught exceptions logged with no context (no user ID, request data, or stack trace)
- Generic 500 responses with no error reference ID for traceability
- Missing try/catch around all external service calls (Supabase, payment providers, APIs)
- No global exception handler customisation in app/Exceptions/Handler.php
- Critical business operations (payments, user creation, data deletion) with no audit log entry
- No structured logging — plain string logs that can't be queried or alerted on
- Missing request/response logging on critical or high-value endpoints
- No alerting hooks on critical exceptions (Slack, PagerDuty, Sentry, etc.) — flag if not configured
- Queue job failures with no notification or monitoring
- Scheduled task failures with no notification
- Any place a user-facing error message could leak internal implementation details

### 8. API Design & Contracts
- Inconsistent HTTP status codes — 200 returned for errors, 500 for validation failures
- Inconsistent response envelope structure — some endpoints wrap in {data:}, others don't
- No API versioning strategy — all routes under /api/ with no version prefix
- Breaking changes risk — any route or response shape change would immediately break clients
- Missing Content-Type: application/json header enforcement on API routes
- Endpoints returning different shapes for the same resource in different contexts
- No standard error response format — clients can't reliably parse errors
- Nullable fields not documented or consistently returned as null vs omitted
- Pagination implemented inconsistently across endpoints (some use page/limit, others offset)
- Any API endpoint with no response type documentation or resource class

### 9. Authentication & Authorisation
- Missing auth middleware on any route that should require login
- Role or permission checks done inconsistently — some use Gates, some use if/else, some use middleware
- Privilege escalation risk — any endpoint where a lower-privileged user could perform a higher-privileged action
- Missing ownership checks — authenticated but not authorised (user can access other users' resources)
- Token scopes not enforced — Sanctum tokens issued with no ability restrictions
- Session fixation risk — session not regenerated after login
- No logout-all-devices / token revocation mechanism
- API tokens stored in localStorage recommended anywhere in frontend-adjacent code (flag as high)
- Missing 2FA enforcement on admin or high-privilege routes

### 10. Data Integrity & Database
- Missing soft deletes on models where hard delete would cause orphaned related records
- Missing cascade or restrict rules on foreign keys — note tables at risk
- Missing unique constraints on columns that should be unique (email, slug, etc.)
- Nullable columns that should not be nullable — flag where null would cause a runtime error
- Missing default values on columns that always require one
- Enum columns in DB not validated against enum values in application code
- Any raw DB::statement() or DB::unprepared() call — flag for review
- Migrations that modify existing columns without a down() method — irreversible migration risk
- Any seeder that could run in production and corrupt live data
- Missing created_at/updated_at timestamps on any model that should be auditable
- Large text or JSON blobs stored in DB that should be in object storage

### 11. Dependency & Supply Chain
- Detect Laravel version from composer.json and flag if not on latest minor patch
- Detect PHP version and flag if below 8.2
- Any composer package with a known CVE — check composer.lock for versions
- Any package that is abandoned or has no releases in 2+ years
- Dev dependencies (phpunit, faker, etc.) present in require instead of require-dev
- Any package installed but not used anywhere in the codebase
- composer.lock missing from version control (if not present, flag as high)
- Any direct git repository dependencies in composer.json instead of packagist

### 12. Testing Coverage
- Scan tests/ directory — note what exists: unit, feature, browser
- Any critical path with no corresponding test: auth, payments, user data mutations
- Tests that make real external calls instead of using mocks/fakes — flag as fragile
- Missing database transaction wrapping in tests — tests that mutate DB state without rollback
- No test for any security-critical endpoint (auth, admin actions, file uploads)
- Feature tests that test implementation details instead of behaviour — brittle test smell
- Estimate overall test coverage based on ratio of test files to application files — flag if low

### 13. Compliance & Privacy
- PII stored without encryption — emails, phone numbers, names, addresses in plaintext
- Any logging that captures PII — request logs, debug logs, error logs containing user data
- Missing data retention policy implementation — old user data never purged
- User deletion that doesn't cascade to all related PII tables — GDPR right to erasure risk
- Passwords or sensitive tokens ever logged, even temporarily
- Third-party analytics or tracking code that sends user data without consent gate
- Any export or report that includes PII accessible to non-admin users
- Missing consent tracking for marketing communications
- Any file upload that stores original filename — can expose user PII or internal paths

### 14. Environment & Deployment Hygiene
- .env.example missing keys that exist in .env (check if .env.example is in version control)
- .env file committed to version control — flag as critical
- Missing production-specific config hardening: cache driver, session driver, log channel
- php artisan optimize not confirmed as part of deployment — config/route/view cache not built
- Missing health check endpoint for load balancer or uptime monitoring
- No queue worker restart strategy after deployment (new code not picked up by running workers)
- APP_KEY rotation — check if it appears hardcoded or committed anywhere
- Any TODO, FIXME, or HACK comment in production code — list all with file and line
- Missing or incomplete .gitignore — any sensitive file types not excluded

### 15. Laravel Best Practices
- env() called directly in code outside of config/ files
- Config values hardcoded instead of using config() helper
- Routes using closures instead of controllers for anything non-trivial
- Missing route model binding — controller manually fetches by ID from request
- Authorization done with manual if statements instead of Gates or Policies
- Any use of Input:: facade (legacy Laravel remnant)
- Controllers over 300 lines — list with line count
- Services over 500 lines — list with line count
- Business logic in routes/web.php or routes/api.php closures
- Deprecated Laravel features for the detected version

### 16. Code Quality
- Unused imports and use statements
- Dead code — unreachable methods, unused classes, commented-out blocks >10 lines
- Missing return type hints on public methods
- Inconsistent naming conventions (methods, variables, routes)
- Duplicate logic appearing in 3+ places that should be extracted
- Magic numbers and strings — unexplained literals that should be constants or config values
- Boolean parameters on public methods (flag/trap pattern) — hard to read call sites
- Methods doing more than one thing — flag any public method over 50 lines
- Deep nesting (4+ levels) — flag for extraction or early return refactor
